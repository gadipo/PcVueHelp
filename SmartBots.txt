Smart Bots blueprint
 

 
 
 
 
 
 
 
 
 
 
 
 
 
Smart Bots blueprint
 
When adding a new Smart Bot, or opening an existing one, the following page is displayed. It allows a user to define the bot states and transitions between these states, to define operations to execute in each state, and configure properties for all items. Show picture
 
The main panel represents all bot states and the transitions between them. The same objects are displayed in the tree view in the left panel. The properties of the selected item are displayed in the Properties Panel.
 
Tools
 
 

 
General tool panel - Allows a user to create a new Smart Bot, print the selected blueprint, open existing Smart Bots, save the current bot or save all opened bots.
 
 

 
Deployment tool panel - Tools to deploy the currently opened Smart Bot to the connected Instant Messaging Server, and to remove it from the server. These tools only work when the Instant Messaging Server is accessible, otherwise an error message appears.
 
 

 
Bot States tool panel - Each bot state is associated with a set of operations executed by the bot agent when entering the state. After executing the operations, the bot agent verifies all outgoing transitions to move to the next state. The Bot States tool panel contains tools to add a new bot state, add a new trigger, create a transition from the selected state, add an operation to the selected state and add conditions to the selected transition or trigger. The Enable all and Disable all tools allow a user to activate / deactivate all operations in the selected state and activate / deactivate all conditions in the selected operation, transition or trigger respectively. Every bot has an initial state.
 
 

 
Debugging tool panel - Tools to start and stop bot debugging. When debugging mode is activated, a special tool panel appears on the screen allowing a user to check the current state and values of variables for all deployed bot agents. The Debugging tool panel also contains tools to add a break-point, Continue the process, and move the process step by step.
 
 

 
Clipboard tool panel - Tools to cut, copy, paste and delete the selected object on the blueprint.
 
 

 
Organize tool panel - Allows a user to arrange bot states and transitions in a visually attractive way. The states and transitions display order has no effect at run-time.

Bot states
 
Each bot state has a single property. This value is referenced in transitions and can be used as the Initial state of the bot.
 
Bot triggers
 
As with bot states, triggers have a Description property. This value is referenced in transitions and can be used as the Initial state of the bot.
 
A trigger contains a set of conditions that are verified on bot agent idle, that is after verifying all outgoing transitions from the current state. Note that, if there is no outgoing transition from the current state, this state is considered as final and the bot therefore proceeds to the completion behavior which is, by default, Return to the end of the last state. This completion behavior may be overridden after the bot entering the trigger state according to the corresponding parameter of the trigger.
 
The following completion behaviors are supported.
 
 

 
Stop procedure. Bot process stops and returns to a caller context if such exists.
 
 

 
Restart from the initial state. Bot restarts the procedure from the initial state keeping the existing memory.
 
 

 
Return to the last state. Bot returns to the last state before trigger has been fired.
 
 

 
Return to the end of the last state. Bot returns to the last state before the trigger has been fired, but not re-executing operations in that state.
 
 
 
The trigger conditions are verified before checking all other conditions of the current state. Thus if the state has a transition with a satisfied condition, the thread will not continue immediately to the following state, but verifies the triggers first. If a trigger also has a satisfied condition, then thread goes to the trigger branch.
 
When a bot enters a trigger branch, it stops verifying other trigger conditions until the trigger branch execution is completed. Giving priority to the execution of a trigger branch avoid uncontrollably switching between two trigger branches when both have satisfied conditions. A trigger branch execution is considered complete when the bot ends up in a final state (a state without any transition out).

Local database changed
 
Monitors changes in the specified local database property. Note that the monitoring is starting only when the smart bot�s process starts to verify the condition. Thus, if the condition is in the trigger, then all database changes are captured. If the condition is in the transition, smart bot will verify only those changes that happened after entering the corresponding state.
 
If the condition is in the trigger, corresponding database changes are monitored and collected in a queue for one hour. Thus if the smart bot detected an appropriate change and started to handle it, and meanwhile the second change has happened, it will be buffered and stored until the smart bot re-verifies the condition again or one hour passes.

Webhook
 
A Webhook condition allows bots to catch and handle external webhooks.
 
The Webhook format is not standard, with each 3rd party system using webhooks defining their own API and payload for the webhook. Therefore there is a set of supported webhook definitions and types. When calling a webhook, a 3rd party system must specify a security code in the Webhook request. The Messaging Web Service ignores all requests without a code or with a wrong code. The correct code is defined in the Messaging Web Service configuration file. Show picture
 

 
 
 
When catching a webhook, the Bot saves its payload to a context that can be extracted using the SaveToMemory operation.

Custom JSON webhook - For Custom JSON webhooks, the Messaging Web Service accepts ordinary get/post HTTP(s) requests on the following endpoint.
https://<host>/Messaging/v2/Webhook?code=<securitycode>&<other query arguments> 
For GET methods query parameters are used as webhook payload. For POST methods the body is expected to be in JSON format which is used as the webhook payload.
 
 

 
Generic JSON webhook - For Generic JSON webhooks the Messaging Web Service accepts post HTTP(s) requests on the following endpoint.
https://<host>/api/webhooks/incoming/genericJson

Stripe - For Stripe webhooks the Messaging Web Service accepts Post HTTP(s) requests on the following endpoint.
https://<host>/api/webhooks/incoming/stripe

Working with data sets
 
Some methods and properties available to a user in the expression return a collection of items. This section describes some methods that can be used to extract a value from the collection as well as calculate some statistical values on the collection.
 
The current Bot Agent context has a method called Historical in order to receive a collection of the variable data points at the specified time range.
 
Returned type of this method is HisDataCollection. After applying Padding method, the collection becomes PaddedHisDataCollection. Both of these types are inherited from IList<HisDataPoint>. This means we can iterate through all data points and compute some function on each data point.
 
In order to pass through all items in the collection and to compute the required value, a LINQ expression can be used. Please refer the Microsoft Help to receive the information about .Net Language Integrated Queries (LINQ).
 
The following list demonstrates some examples of getting the Historical data set.
 
 

 
Historical data points during the last 1 day.
 
x => x.Historical("Register01", "Unit01", new DateRange(1, Intervals.Days))
 
 

 
Historical data points during the last 4 hours.
 
x => x.Historical("Register01", "Unit01", new DateRange(4, Intervals.Hours))
 
 

 
Historical data points between 01/01/2019 and 31/01/2019.
 
x => x.Historical(�Branch .Register01�, �Unit01�, new DateRange(new DateTime(2019, 1, 1), new DateTime(2019, 1, 31)))
 
 
 
The following list demonstrates some examples of getting the padded historical data set.
 
 

 
Historical data points with padding of 1 second (1 data point every second).
 
x => x.Historical("Register01", "Unit01", new DateRange(1, Intervals.Days)).Padding(1, Intervals.Seconds)
 
 

 
Historical data points with padding of 1 minute (1 data point every minute).
 
x => x.Historical("Register01", "Unit01", new DateRange(1, Intervals.Days)).Padding(1, Intervals.Minutes)
 
 

 
Historical data points with 1 minute padding, when Bit01 was activated.
 
x => x.Historical("Register01", "Unit01", new DateRange(1, Intervals.Days)).Padding(1, Intervals.Minutes).Filter(�Bit01�)
 
 

 
Historical data points with 1 minute padding, when both Bit01 and Bit02 were activated.
 
x => x.Historical("Register01", "Unit01", new DateRange(1, Intervals.Days)).Padding(1, Intervals.Minutes).Filter(�Bit01�) .Filter(�Bit02�)
 
 

 
Historical data points with Good quality and with 1 minute padding, when both Bit01 and Bit02 were activated.
 
x => x.Historical("Register01", "Unit01", new DateRange(1, Intervals.Days)).Padding(1, Intervals.Minutes).Filter(�Bit01�) .Filter(�Bit02�).ExcludeNS()

Bot agent memory
 
Every bot agent has an internal memory with a set of key-value pairs. The operation SaveInMemory can be used to add or replace an item in the memory.
 
Almost every text property of bot conditions and operations support substitution of values stored in memory, called internal bot variables.
 
In order to reference such value, the following notation is used: {key}. In this case, the {key} string will be replaced with the value stored in memory before verifying a condition or executing an operation.
 
In the following example, State1 is associated with a set of operations as follows. Show picture
 

 

 
 
 
 
 

 
The first operation creates a text internal bot variable with the value of the last written text message
 
 

 
The second operation puts the Sender of the last message to the variable send
 
 

 
The last operation sends a text message to the current chat. Note that the text is defined as Hi, {send}, where the {send} will be substituted with the value of the send variable, that is the Sender of the last message. For example, Hi, Ope1 will be sent to the chat in response to the message from Ope1.

Smart Bots operations overview
 
The following categories of operations can be used in a bot.
 
 

 
Process
 
 
 

 
Data - Remote, Local, Web and Prediction

Messaging

Access rights
 
 
 

 
Schedule
 
 
 
 
To create a new operation, select the bot state to which the operation will be added. Then right click to open the context menu, or use the New operation tool in the tool bar. Select the required operation in the drop-down menu. Show picture

 
Operations are executed sequentially. The order of operation can be changed using the tools Move up (Alt+Up arrow) and Move down (Alt+Down arrow).
 
An operation can be associated with one or several conditions that must be satisfied for the operation to be executed. Conditions are verified only once before deciding whether the operation will be executed or not. This means that Value Changed conditions will never be satisfied and thus an operation containing such conditions will not be executed.

Smart Bots overview
 

 
 
 
 
 
 
 
 
 
 
 
Smart Bots overview
 
Smart Bots is a plug-in for the Instant Messaging Server. It allows the application designer to create and run automated programs, called bots, which execute a set of operations in response to fulfillment of conditions. Examples of the use of Smart Bots includes the following.
 
 

 
Application forms.
 
 

 
Job orders.
 
 

 
Field Service Management.
 
 

 
On-site inspections work flow.
 
 

 
Smart Assistants.
 
 
 
Smart Bots are designed to automate recurrent daily tasks, currently executed by operators and administrators manually. Describing and formalizing the process in terms of parametrized states and transitions between them allows defining flexibly nearly any existing process.
 
Deployment
 
The Smart Bots plug-in is available as part of PcVue's installation. To install it select the Custom installation and select the Smart Bot feature.
 
 

 
Bots are loaded from the folder <projectFolder>/MsgData/Bots. Plugin and bots are started along with the Instant Messaging Server.
 
 

 
Bots store the local data in the folder <projectFolder>/MsgData/Bots/Data.
 
 

 
Schedules are loaded from the folder <projectFolder>/MsgData/Schedulers. Scheduler is started along with the Instant Messaging Server.
 
 

 
The bots configuration file is loaded from the folder <projectFolder>/MsgData/C. This configuration is used by Smart Bots at run-time for corresponding operations.
 
 
 
Smart Bots nomenclature
 
 

 
Smart Bot - A process blueprint described by a finite-state machine with a set of operations in each state and conditions defined for every transition. Sometimes abbreviated to just Bot.
 
 

 
Bot agent - A thread running the given Smart Bot in the given chat. An instance of Bot agent is created for every chat where this Smart Bot has access to. A bot agent runs a finite state machine with defined states and transitions between states. Every Smart Bot has an initial state. When the bot agent arrives to a defined state, a corresponding set of operations is executed. After that, a set of conditions is continuously verified. If a certain condition is met, the bot will pass to the next state following the transition.
 
 

 
Trigger - A special state containing conditions instead of operations. Triggers are only verified when the bot agent is idle, that is when it is running but cannot move forward on any transition. If all conditions of a trigger are met, the bot will pass to the trigger state. After the triggered scenario is finished, that is when bot agent arrives to the final state in the trigger branch, the bot will stop, return to the last state or reset from the initial state depending on the specified behavior.
 
 

 
Scheduler - A thread running at the Instant Messaging Server process. It monitors the loaded schedules and execute Bot agents at the planned time.
 
 
 
Bot permissions
 
An independent bot agent is started for each chat that the bot has access to. This implies two cases:
 
 

 
The bot agent is started in a chat when the bot is invited to the chat.
 
 

 
The bot agent is started in a chat if the bot belongs to a super profile (with a profile level greater than the level of all chat participants).
 
 
 
Each bot agent has permissions according to its role in the chat. For example, if the bot agent is invited as a guest, it will be able to send and read messages, but will not be able to invite other users to the chat. See Instant Messaging book for further details on user permissions.
 
The bot has the same set of permissions as any other user with the same role.

Smart Bots types
 

 
 
 
 
 
 
 
 
 
 
 
 
 
Smart Bots types
 
The smart bots property Bot Type defines how the bot is handled on start-up and during development.
 
Smart bots
 
Bots of type Smart Bot are bots designed to interact in a chat room. These bots are loaded from the �<projectFolder>/MsgData/Bots� folder at the Instant Messaging Server start-up. Bots of this type must be added to the chat room in order to be started.
 
System bots
 
System Bots are designed to work in the background without any user interaction. When deploying a System Bot, the Instant Messaging Server automatically creates a chat room for it. The following values are used for the chat properties.
 
 

 
Id - The bot�s Id.
 
 

 
	Name - The bot�s name.
 
 

 
Description - The bot�s description.
 
 

 
Color - A random color from the list of available colors.
 
 

 
Storage time - One day.
 
 

 
Associated Zone - None.
 
 
 
The deployed bot is automatically invited to this chat room as the owner and therefore gets started.
 
Assistant Bots
 
Assistant Bots are designed to work one to one with a user. The configuration specifies which users, profiles and profile roles it will serve. Usually these bots are used to provide a specific HMI to mobile users. An assistant bot may be associated with the following sets of users:
 
 

 
A set of users. 
 
 

 
A set of user profiles.
 
 

 
A	set of profile roles.
 
 
 
Multiple users / profiles / profile roles can be defined by separating them with a comma or semicolon. Multiple users / profiles / profile roles can also be defined using wild-card notation with * corresponding to any amount of any characters and ? corresponding to any single character.
 
When deploying an assistant bot, the Instant Messaging Server automatically creates a chat room for every associated user. If the profile is specified, the chat room is created for each user with the given profile.
 
The following values are used for the chat properties.
 
 

 
Id - The bot�s Id.
 
 

 
Name - The bot�s name.
 
 

 
Description - The bot�s description.
 
 

 
Color - A random color from the list of available colors.
 
 

 
Storage time - One day.
 
 

 
Associated Zone - None.
 
 
 
The deployed bot is automatically invited to these chat rooms as the owner and therefore gets started.
 
Template Bots
 
Template bots are custom parametrized operations for use by other bots. A template bot is not visible in the chat interface and therefore cannot be added to a chat room. A template bot has a set of argument / parameter definitions.
