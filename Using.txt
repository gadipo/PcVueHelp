Using a 3D camera model
 

 
 
 
 
 
 
 
 
 
 
 
 
 
 Using a 3D Camera Model
 
A camera model is a 3D model that is either custom or
 built-in (the HMI offers the choice). When it is inserted in a window,
 the HMI automatically memorizes the current settings of the window camera. These settings
 can later be re-applied to it, allowing an instantaneous transition to
 a different view of the 3D world. You can modify (or update) the settings
 memorized by a camera model. 
 
The types of camera model available are as follows.
 
 
Custom camera model
 - Any of the 3D models located in the project libraries.
 
Built-in camera model
 - A 3D model provided by the HMI and located in the HMI's resources. It
 is not available as a standard model (i.e. located in a library).
 
 
Except for the way they are inserted in a window, camera
 models of both types are processed identically by the HMI. 
 
Inserting a camera model
 
You can insert a camera model as follows: 
 
 
From the Insert menu,
 select one of the Insert Camera commands.
 
From the 3D Draw
 toolbar, select one of the Insert Camera commands. Show picture

 
 
When you insert a custom camera, the 3D model library
 is automatically opened for selection of the model that will represent
 the camera. However, when you insert a built-in camera, the built-in 3D model is
 directly inserted in the mimic.
 
Several camera models can be inserted in a single window.
 

 
When a camera model is inserted, it is usually (depending
 on the model shape) not visible because it is positioned at the current
 location of the window camera.
 
When a camera model is inserted, that model is not automatically
 selected.
 
 
Displaying a camera model
 
Unlike a standard 3D model, when a camera model is selected,
 the HMI does not show the model axes. This is because the HMI does not
 allow for moving, rotating or scaling a camera model directly in the mimic.
 
The parameters for camera model position and axes rotation
 are automatically calculated using the camera setting parameters and using
 the model's Z axis as the aiming axis of the camera. 
 
This picture shows a built-in camera model that is selected.
 Show picture
 

 

 
 
 
 

 
When you use an orthographic view, the camera-aiming axis
 is always parallel to one of the world axes and always perpendicular to
 the orthographic projection plane (defined by the two other axes of the
 3D world).

Using automatic logoff
 

 
 
 
 
 
 
 
 
 
 
 
 
 
Using Automatic Logoff
 
Automatic logoff automatically logs off the user after a period of inactivity. Inactivity is defined as the time since the last use of any user input device (normally the keyboard and/or mouse).

 
Automatic logoff can be tuned based on 2 settings.
 
 
In the HMI Options dialog box. This enables or disable the automatic logoff for the entire project. By default, it is enabled. You can also enter the period for which the warning dialog is displayed.
 By default, this is set to zero.
 Show picture
 

 

 
 

 
 
In each user profile, you can enable/disable the automatic logoff and the max inactivity period on a per profile basis. See the topic Profile Properties that Affect Behavior When Logging On and Off.
 
 
How the automatic logoff works

 
Automatic logoff takes place in two stages.
 
 
After a period of inactivity equal to that specified in the User's profile the Automatic logoff dialog is displayed. 
 Show picture
 

 

 
 

 
 
The Automatic logoff dialog initially displays the number of seconds that was entered in the HMI Options dialog. The seconds count down to zero at which point the User is logged off and the dialog closes.

 
 
Any activity will reset both timers and close the Automatic logoff dialog (if open) and the process starts again.

 

 
If the default period of zero is used (in the HMI Options dialog), no warning dialog is displayed before the User is logged off.
 
Automatic logoff is supported in WebVue, but no warning box will be displayed to the inactive user.

In any animation where a user enters a value at run-time.

Using drawing elements in a VBA script
 

 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
Using drawing elements in a VBA script
 
Before any HMI drawing element (Shape, Text, Image etc.) can be used in a VBA script, you must select VBA control in the pop-up menu displayed by right-clicking on the object.
 
Show picture
 
 

 

 
 
 
 
Once the drawing element has been enabled as a VBA control, it becomes visible in the VBA environment and its EnableEvents property is set to True.
 
 
It can be selected using the left-hand combobox in the code window
 
Once it is selected in the left-hand combobox, you can select any of its events in the right-hand combobox and directly code the event
 
It is visible in the VBA property box
 
The object is directly known by its name (for example 'shape1') in the VBA Project
 
 
You can de-select the VBA control property at any time. The associated code is not lost if you de-select a control, it just becomes a function of the Project.
 
It is possible for a drawing element to be a VBA control without the EnableEvents property being True. In this case, you will be able to change its properties by script, but not be able to run code on event.
 
Each drawing element that is enabled as a VBA control requires additional resources from the host computer. Only enable drawing elements as VBA controls if you need to develop code for them.
 
Making drawing elements respond to the mouse pointer
 
This topic describes how to make drawing elements respond visually when the mouse pointer is held over them.
 
Show picture
 
 

 

 
 
 
 
How to make a drawing element respond to the mouse pointer:
 
 
Select a drawing element and open the Properties List (View.Properties List)
 
Select the DisplayMouseInput property and set it to True. Click on the value, click on the button marked '...' and select True
 
 
If set as described above, when you hold the mouse pointer over the drawing element, it responds as if it had a Send animation. When elements overlap, the effects depend on the combination of the DisplayMouseInput property and configured animations.
 
 
 
 

 Elements
 Effect
 
 

 One element with the property True
 It responds all over.
 
 

 
 
Two elements:
 
 

 
one with the property True
 
 

 
one with an animation
 
 
 
 Each element responds separately except that when the pointer is over the overlap, both respond.
 
 

 Two or more overlapping elements, both with the property True (or both with animations)
 Each element responds separately. Only the one in front responds in the overlapping area.
 
 

 Three or more overlapping elements.
 They respond separately except that when the pointer is over the overlap of a particular pair - one with the property True, the other with an animation - that pair will respond together.

Using the workspace regions
 
The workspace can be divided into up to four equally spaced 
 horizontal or vertical regions. When opening a window you can then specify the region 
 in which it is opened. Regions are most commonly used when running PcVue on a PC that has multiple screens when you would create as 
 many regions as you have screens. A window can then be opened on a specific 
 screen.
 
The number of regions is selected using the Regions tab 
 of the workspace properties dialog. Show picture
 

 

 
 
 
 
How to configure the workspace regions
 
 
Right click anywhere 
 in the workspace and select Workspace 
 Properties from the pop-up menu. The Workspace properties dialog 
 is displayed.
 
Select the Regions 
 tab.
 
Using the Region 
 system drop down list box select the number of regions.
 
Using the Orientation drop down list box select the region arrangement as either vertical or horizontal.
 
Select the Startup region by clicking on the corresponding icon. The startup region 
 is the default region in which a window is opened if no region has been 
 specified.
 
 
What is the difference between Screen or Workspace 
 resolution
 
 
Screen resolution 
 - Determines the size and location of the regions according to the resolution 
 of the entire Windows' desktop.
 
Workspace resolution 
 - Determines the size and location of the regions according to the resolution 
 of the PcVue workspace.

Using a SQL Expression to Filter the Variables
 
A SQL expression can be used in the Grid control and the Alarm Viewer to 
 filter the list of displayed variables or alarms. It can also be used to set a filter via a SCADA Basic script using the instructions SELECTOR, ALARMDISPLAY and LOGDISPLAY.
 
The SQL expression takes the following 
 form: <Criteria> <Operator> <Value> 
 
Criteria
 
Each of the criteria used in the expression represents 
 one of the properties of a variable.
 
 
 
 
 

 
 
Name
 
 
 
Data type
 
 
 
Description
 
 
 

 
 
Name
 
 
 
String
 
 
 
Variable name.
 
 
 

 
 
Domain
 
 
 
String
 
 
 
Domain name.
 
 
 

 
 
Nature
 
 
 
String
 
 
 
Nature name.
 
 
 

 
 
AssocLabel
 
 
 
String
 
 
 
Associated Label name. Applies to Bit and Alarm variables only.
 
 
 

 
 
AlarmLevel
 
 
 
Integer
 
 
 
Alarm level value. Applies to Alarm variables only.
 
 
 

 
 
BinAttr
 
 
 
Long
 
 
 
Binary attributes as weighted bits.
 
 
 

 
 
BinAttrbNN
 
 
 
Boolean
 
 
 
Individual binary attribute where NN is in the range 01 to 30.
 
 
 

 
 
TextAttrNN
 
 
 
String
 
 
 
Text attribute where NN is in the range 03 to 16.
 
 
 

 
 
TextIndAttrNN
 
 
 
String
 
 
 
Indirect value of the text attribute where NN is in the range 03 to 16. The indirect value is the value of the variable whose 
 name is contained in the text attribute.
 
 
 

 
 
Description
 
 
 
String
 
 
 
Variable description. The description criteria is language 
 dependent.
 
 
 

 
 
Unit
 
 
 
String
 
 
 
Variable unit text. Applies to register variables only.
 
 
 

 
 
Format
 
 
 
String
 
 
 
Variable display format. Applies to register and text variables only.
 
 
 

 
 
Threshold
 
 
 
Double
 
 
 
Threshold. Applies to alarm and bit variables defined as threshold of a register variable.
 
 
 

 
 
LoValueLimit
 
 
 
Double
 
 
 
Minimum value. Applies to register variables only.
 
 
 

 
 
HiValueLimit
 
 
 
Double
 
 
 
Maximum value. Applies to register variables only.
 
 
 

 
 
Simulated
 
 
 
Boolean
 
 
 
Variable simulated status.
 
 
 
 

 
Criteria names are not case sensitive. 
 
 
Operators
 
The following operators are available for the construction 
 of the expression that compares the value of a variable property (criteria) 
 to a user-defined value.
 
 
 
 

 
 
Operator
 
 
 
Description
 
 
 

 
 
<
 
 
 
To check if a property value is smaller than a user-defined value.
 
 
 

 
 
>
 
 
 
To check if a property value is greater than a user-defined value.
 
 
 

 
 
<=
 
 
 
To check if a property value is smaller than or equal to a user-defined value.
 
 
 

 
 
>=
 
 
 
To check if a property value is greater than or equal to a user-defined value.
 
 
 

 
 
<>
 
 
 
To check if a property value is different from a user-defined value.
 
 
 

 
 
=
 
 
 
To check if a property value is equal to a user-defined value.
 
 
 

 
 
Like
 
 
 
To check if a property string value is equal to a user-defined string value using wild characters. See the following section on values.
 
 
 

 
 
Unlike
 
 
 
To check if a property string value is different than 
 a user-defined string value using wild characters. See the following section 
 on values.
 
 
 

 In
 To check if a property value is within a list of values. The list of values must be comma separated and contained within brackets.
 
 
 
Expressions can be combined by using parentheses and the following logical operators.
 
 
AND - Combines two expressions where both expressions must be true for the result to be true.
 
OR - Combines two expressions where either or both expressions must be true for the result to be true.
 
 

 
String comparisons are not case sensitive. 
 
 
Values
 
The user defined values, to which the property values are compared, can be a string, a Boolean or a numeric value, depending on the property data type.
 
 
Boolean - The value can be 0, 1, true, false.
 
Integer - Any numeric value that can be represented by an integer and that is in the range of the property minimum and maximum limits.
 
Double - Any numeric value that can be represented by a double and that is in the range of the property minimum and maximum limits.
 
String - Any string value delimited by leading and ending double quotes.
 
 
When used with the LIKE or UNLIKE operator, a string value can also contain the following wild card characters.
 
 
? - Represents any single character. A string value can contain several question mark characters, consecutively or not.
 
* - Represent any number of characters.
 
 
Using the Language property
 
The language property is used to select the language 
 that the filtering process uses when executing string value comparisons. 
 The combo-box lists the base and alternate languages and also provides 
 a current option referring to the language selected at the time the filter 
 is executed. If the project is not configured as bilingual then the Language 
 property is not shown.
 

 
Up until version 8.10, only the Description property is bilingual.
 
 
Examples
 
 
Name='Building1.Room2.Lights' - Will display the variable with the name "Building1.Room2.Lights".
 
Name Like 'Building1.*' - Will display all the variables that have a name beginning with "Building1.".
 
Name Like 'Building?.*.Lights' - Will display all the variables that have a name beginning with "Building" immediately followed by any one and only one character and ending with ".Lights".
 
Name Unlike 'SYSTEM.*' - Will display all variables except those starting with SYSTEM (Excludes all system variables).
 
BinAttrB01<>False - Will display all the variables which the binary attribute 1 is different from False.
 
LoValueLimit >= 0 - Will display all the register variables that have a low limit value greater than or equal to 0.
 
(Name Like 'B*' 
 OR Name Like 'M??') AND Simulated = False - Will display all 
 the variables that have a name beginning with "B" or a name 
 beginning with �M� immediately followed by any combination of two characters 
 and not defined as simulated variables.
 
AlarmLevel In (1,3,5) - Will display all variable with an alarm level of 1, 3 or 5.

Using a variable's extended attributes in an animation
 
You can use a variable's extended attributes in an appropriate animation 
 by appending the attribute's mnemonic to the variable name. By appropriate we mean an animation expecting a text value for a text attribute, or an animation expecting a bit value for a binary attribute.
 
 
To display a text attribute append .TATTn# to the variable name where 
 n is the number of the text attribute. Note that TATT1# and TATT2# are 
 the variable's Domain and Nature respectively.
 
To display a binary attribute append .BATTn# to the variable name where 
 n is the position of the bit within the binary attribute. For example, 
 EFFLUENT.TANK2.STIRMOTOR.RUNNING.BATT2# would display a 1 if the binary 
 attribute were 00000010.
 
 
Displaying the timestamp of a variable
 
The timestamp of a variable is the time at which the value last changed. To display the timestamp you append .#_T to the variable name in the Display Register animation. For example, to display the timestamp for TANK1.VALVE1 you would use TANK1.VALVE1.#_T. To display the timestamp correctly you must use the Custom Display Format option and select DATE for the received unit and the appropriate format for the output unit. For example, hh:nn:ss dd/mm/yy@t Show picture
 

 
 
 
Displaying the quality of variable
 
The quality of a variable provides an indication of the value quality. It is mainly of use when the variable's source is an OPC server. To display quality you append .#_Q to the variable name in the Display Register animation. For example, to display the quality of TANK1.VALVE1 you would use TANK1.VALVE1.#_Q
 
Using the simulation properties of a variable
 
The properties of a variable related to simulation can be used in an animation:
 
 

 
To display the value of the Simulated property, you append .SIMU# to the variable name in any animation based on a Bit variable. The property can be used as the base for a display animation to indicate whether the variable is simulated or not (color on bit for example), and it can also be used to set/reset the variable's Simulated property using a Send bit animation.
 
 

 
To display the fallback value property, you append .SIMFALLBACK# to the variable name in an animation of the proper type (bit, register or text animation depending on the simulated variable type). The property can be used as the base for a display animation to display the fallback value, and it can also be used to set it using a Send animation.
 
 

 
To display the expiration time property, you append .SIMEXPDATE# to the variable name in a Register animation. The property can be used as the base for a display animation to display the expiration time, and it can also be used to set it using a Send Register animation.

 
 

 
Use a custom format to display the simulation expiration time as a date and time.

Controlling instantiation using tagging
 

 
 
 
 
 
 
 
 
 
 
 
 
 
Controlling Instantiation Using Tagging
 
Tagging can be used with the following configuration elements.
 
 
Symbols to be included with Mimics
 
File Items to be included with Files.
 
 
Consider the following example for mimic and symbol generation
 
Templates have been created to represent the floor of a building (FLOOR), the offices within that floor (OFFICE), and the lights within the office (LUM). LUM generates an instance of a symbol, OFFICE and FLOOR both generate mimics. The template structure looks like this. Show picture

 
The requirement is that the symbol representing the lights (generated by LUM) is only displayed in the mimic generated by OFFICE and not in the mimic generated by FLOOR. Without filtering, there will be two instances of the light symbol in each of the OFFICE mimics and all six in the FLOOR mimic.
 
How to configure tagging
 
Tagging works by comparing the Tags property of the symbol with the Tags to include property of the mimic. If there is a match then the symbol is instantiated in the mimic. Taking the above example, the Tags and Tags to include properties for both the OFFICE mimic and the LUM symbol is fOffice. The Tags to include property for the FLOOR mimic is fNone. As the Tags property of the symbol for LUM matches the one of the mimic for OFFICE, the symbol for LUM will be instantiated in the mimic for OFFICE. As it does not match the Tags to include property of the mimic for FLOOR, no instantiations of the symbol for LUM will be made in the mimic for FLOOR. Show picture

 

 
Multiple tag strings can be defined by separating them with a semi-colon. For example, fOffice;fStairwell;fEntrance;fMaintenance. It allows generating multiple mimics with different symbols for different purpose. For example, one may need to have a monitoring and control mimic for each floor, including symbols designed for this purpose, and a maintenance mimic for each building, with another dedicated graphic representation (another symbol), but for the exact same lighting devices (and therefore defined in the same template).

Using the color animations with 3D models
 

 
 
 
 
 
 
 
 
 
 
 
Using the Color Animations with 3D Models
 
The color of a 3D model is determined by the model's 
 materials. A color animation, when applied to a 3D model, changes properties 
 that are supplied by the materials - it does not change the materials 
 themselves. For example, it can change the emissive color for a mesh, 
 but it does not change the configuration for that color in the material 
 used to color the mesh.
 

 
Throughout this book you will see references to animating of materials. This term is used to simplify the explanation. What you are 
 actually doing is animating the color or other property of a mesh that, 
 if it were not animated, would be supplied directly by the material.
 
 
What material properties can be changed?
 
A single color animation can change any or all of the 
 following material properties.
 
 
Diffuse color
 
Opacity (percentage)
 
Specular color
 
Specular level (percentage)
 
Emissive color
 
 
How many color changes can I make?
 
When configuring a color animation you can select one 
 or more materials. The same color changes are applied to all selected 
 materials. The number of color changes that you can make with a single 
 animation depends on the animation type. For more information, see the 
 book on Color Animation in the main HMI Help.

Level
 
 
 
The color animation is applied to
 
 
 

 
 
Symbol or group
 
 
 
All materials of all models in the symbol or group.
 
 
 

 
 
Model
 
 
 
One or more of the materials depending on how the animation

Frame without a mesh
 
 
 
All materials used by its sub-frames.
 
 
 

 
 
Frame with a mesh
 
 
 
One or more of the materials used by the mesh and/or all 
 materials used by any sub-frames.

Using variables in an animation
 

 
 
 
 
 
 
 
 
 
 
 
Using Variables in an Animation
 
To use a variable in an animation you enter its name 
 optionally preceded by the @ character. If you already know the name, you 
 can enter it directly but it is normally easier to use the Variable Browser 
 to select the name. The following are all examples of valid variable names.
 
Process.Engine1.Compressor1.B1
 
@Process.Engine1.Compressor1.B1
Process.Engine1.Compressor1.A1
CARD1.VARIABLESTREE.B1
Node1.DataTable3_256
Simulation.Square Waves.Int4
B1
 
For further information, see the topic Using the Variable Selector.
 
Using relative referencing
 
If the animation is in a window that has been opened 
 with a branch (for example Process.Engine1) you can use relative referencing. 
 With relative referencing, each variable reference uses only part of the 
 variable name. For example, you might use the name Compressor1.B1. The 
 other part of the variable name is supplied by the branch of the window. 
 The variable names are automatically calculated at run-time by joining 
 the branch with the relative name in the animation.
 
When using the Variable Browser to select a variable 
 in a window with a branch it automatically selects the correct part of 
 the variable name for you. For example, if the window was opened with the 
 branch Process.Engine2 and you selected the variable Process.Engine2.Compressor1.B1 
 it would insert Compressor1.B1.
 
Using the ?.VarName syntax
 
Unless prefixed with the @ character, variable names that you include in an expression are relative to any branch being used. However, it is possible to reference variables starting back from the root of the branch hierarchy by using the ?.VarName syntax. For example, the expression is used with the branch ProcessB.Machine2.Engine1 but you need to use the variable ProcessB.NumRunning. In that case you can use ?.NumRunning in the expression. The question mark tells to start from the root level in the branch hierarchy when resolving the variable name, and in this example, it would result in the variable name ProcessB.NumRunning being used in the expression at run-time.
 
You can use more question marks to address all intermediate levels in the branch hierarchy. For example, ?.?.?.VarName would go down three levels from the root. 
 
This technique can also be used when designing symbols using relative referencing. See the topic Creating a relative referencing symbol.
 
Forcing direct referencing
 
Sometimes in a window that has been opened with a branch, 
 you need to refer to a variable with a different branch or no branch at 
 all. To achieve this you use direct referencing where the variable name 
 is preceded by the @ character. For example, @CARD1.VARIABLESTREE.B1. When using 
 direct referencing the same variable is always referred to whichever branch 
 the window is opened with. If you use the Variable Browser to select the 
 variable the @ character is inserted automatically when you select a variable 
 with a different branch to that of the window.
 

 
The name of a variable with the prefix @ is known as the full name of this variable. When a variable is referenced in such a way in an animation, the reference is direct and any branch is ignored.
 
 
Errors when selecting variables
 
The following error icons may appear alongside the support 
 drawing element when you select a variable for an animation.
 
 
 
 

 
 

 
 
 
 
 
The variable does not exist.
 
 
 

 
 

 
 
 
 
 
The variable exists but it has a bad status (for example 
 - out of limits). Depending on the HMI option Variable warning display, the icon with an exclamation mark may be used instead.
 
 
 

 
 

 
 
 
 
 
The object could not be initialized. This icon appears only for objects such as:
 
 
ActiveX controls - Most probable cause is that the ActiveX is missing or not correctly registered on the computer.
 
.Avi files - If the file cannot be found.

Using variables in a VBA script
 

 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
 
Using variables in a VBA script
 
Before you can use the properties, methods and events of a variable configured in PcVue in a script, the variable must first be subscribed.
 
Variables are subscribed in the following ways:
 
 
Automatically when used in an animation of any open mimic
 
Automatically whenever a variables value is referenced in a VBA program with the [] syntax
 
Manually by using the Add method of the Variables collection
 
 
Subscribing a variable
 
Any variables used in any animation in an opened mimic are automatically made available to VBA. It is unnecessary to programmatically subscribe to the variable as described below. However, when the last mimic using a particular variable in an animation is closed, that variable is no longer available to VBA unless it is (also) programmatically subscribed.
 
To subscribe a variable by code, you use the Variables.Add method.
 
For example:
 
Variables.Add "Branch1.B1", fvVariableTypeBit
Variables.Add "CommentDisplay%", fvVariableTypeText
 
The optional second parameter describes the variable type. For more information, see the Variables Collection topic.
 
If the variable is already subscribed by an animation in the HMI, it may still be subscribed explicitly through VBA. When the mimic containing the animation is closed, the variable will then remain subscribed until the project is closed or it is unsubscribed in VBA.
 

 
You must use the full variable name when you subscribe a variable.
 
 

 
Some variables may need to be subscribed 'permanently' while the project is running. For example, variables for which a script must be executed upon value change, independently from open mimics. You can subscribe to these variables in the call-back script of the event StartupComplete, and unsubscribe in the call-back script of the event BeforeQuit.
 
 
Unsubscribing a variable
 
To unsubscribe a variable you use the Variables.Remove method.
 
For example:
 
Variables.Remove "Branch1.B1"
 
If you subscribe a variable more than once in VBA, you have to unsubscribe it that many times before the variable can become unsubscribed. Even then, it remains subscribed in the HMI so long as any mimic that uses it in animations remains open.
 

 
Each variable that you subscribe uses some slight system resources. It is recommended to unsubscribe a variable as soon as it is no longer needed.
 
 
Shortcuts for accessing variables in VBA
 
There is a variety of syntax shortcuts to address a Variable and the Variables collection.
 
The Item method
 
The Item method is the default for the Variables collection, and the Value property is the default for the Variable object. The default method and propety of an object can always be omitted. So:
 
Variables("CommentDisplay") = "Hello World"
 
is the equivalent of:
 
Variables.Item("CommentDisplay").Value = "Hello World"
 
The [...] syntax
 
The [...] syntax provides an additional shortcut to directly access a variable's value.
 
[Main.Branch1.B1] 
 
is the equivalent of 
 
Variables("Main.Branch1.B1")
 
It is recommended that you use this shortcut to make your code more readable.
 

 
When you use the [...] syntax, the variable is automatically subscribed and unsubscribed - there is no requirement to do it programmatically earlier in your code.
 
 
Example:
 
Private Sub Shape1_Click()
 If [Main.Branch1.B1] = True Then
 &#160;&#160;[Main.Branch1.B1] = False
 Else
 &#160;&#160;[Main.Branch1.B1] = True
 End If
End Sub
 
Running VBA code when a variable's value changes
 
You can execute code when a variable's value changes, using the Dim WithEvents statement and the ValueChange event.
 
This is best illustrated with a simple example:
 
'------ This code must be placed in General Declarations
Dim WithEvents objVar As Variable

'------ Execute to set up events
Private Sub fvProject_StartupComplete()
'------ Subscribe to variables
 Variables.Add "Branch1.B1", fvVariableTypeBit
' ------ Assigns an object reference to the variable and enable the events
 Set objVar = [Branch1.B1]
 objVar.EnableEvents = True
End Sub

'------ Execute when value of Main:Branch1.B1 changes
Private Sub objVar_ValueChange()
 MsgBox objVar.Name & " = " & objVar.Value
End Sub
 

 
You must declare the Variable object using the WithEvents keyword and enable events for the object by setting the EnableEvents property to True.
 
 
If the code is contained within a mimic, any references to objects are temporary. When the mimic is closed, or its mode changed (Run/Design), the reference is deleted and the event no longer fired. For the event to be fired again, the object must be reassigned to the variable and the events enabled again.

Using mimics with a branch
 

 
 
 
 
 
 
 
 
 
 
 
Using Mimics with a Branch
 
When a mimic is opened or created, it may be given a variable 
 branch dynamically at opening time. Subsequently any reference to variables is relative to that branch. 
 This technique can 
 be used to reduce the number of mimics required and the subsequent development 
 time.
 
An example of using a mimic with a variable Branch
 
Consider an application where there are four identical 
 gas turbines to be supervised. Each turbine has more than 100 variables 
 associated with it. In PcVue, each variable name starts with the 
 turbine number.
 
@turbine1.speed, @turbine1.pressure etc.
@turbine2.speed, @turbine2.pressure etc.
@turbine3.speed, @turbine3.pressure etc.
@turbine4.speed, @turbine4.pressure etc.
 
A mimic named Turbine is created initially with the Branch turbine1. Subsequently all variable references are relative to the branch turbine1, 
 so speed actually refers to @turbine1.speed. A further mimic named 
 Menu is created with four buttons each one with an animation to open the 
 Turbine mimic but using four different Branches each corresponding to 
 one of the turbines (turbine1, turbine2, turbine3 and turbine4).
 
At runtime, the Turbine mimic may be opened four times, 
 each time displaying a different group of variables.
 
With this example, instead of creating four mimics it 
 was only necessary to create one. This has the dual advantage of reducing 
 development time and making maintenance much easier.
