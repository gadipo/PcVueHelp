I/O and internal variables - HMI variables

See also

All variables are available to all programs.

If the value of a variable is to be changed from within a program, then the variable must have the Control attribute set.

It must not be made globally in the program header nor passed as an argument from a Run Program animation. See example in the DIM instruction's topic.

Variables may be referenced directly using their full name, or by using the branch instruction and then using a name relative to the branch. A working variable of type string may also be used to defer the resolution of the variable name.

When referring to variables within a program, you must ensure that your program manages the variable type correctly as in the table below.

Variable

Type in program

Range

Up to 32,000 characters.

In early versions of PcVue a register was treated as type single. This was changed so that a register is treated as type double. For compatibility purposes the following features were added at the time of change.

  • A register can be used as either type double or type single. Using as type single may result in loss of precision.

  • A property, Use register variables as double in SCADA Basic, was added to Programs Settings. This is set to True by default but if changed to false enforces strict use of a register as type single.

Some of the code examples were written before a register was treated as a double and therefore may contain unnecessary type conversions.

Using Deferred Naming

When referring to variables in a program, you can hard code the variable name as part of the program. Sometimes it is better to construct the name of the variable in the program itself so that the resolution is deferred and only performed at execution time, for example by passing the variable name as an argument, or calculating the name programmatically. When using this technique the variable name is stored in another (string) variable, usually a working variable. This is known as indirect or deferred naming and uses the "?" sign.

You can only use deferred naming in the operands of a value assignment with the = operator. It can be used as part of the left-side or right-side operands. It will not work in any other context. See the examples below.

If deferred naming is used, the working variable that contains the name of the variable must be declared inside the SUB function where it is used. See the examples below.

To refer to the variable, you use the name of the string preceded by a question mark "?". This is best illustrated by an example.

SUB Init ()
 
    DIM VarName As Str;
 
    DIM Index as Integer;
 
 
 
    For Index = 0; Index < 11; Index ++
 
        VarName = AddString("Tank", TOC(Index));
 
        Varname = Addstring(Varname, ".Level.SP");
 
        '-------- Deferred variable referencing
 
        ?Varname = 0;  '? used as part of the left-side operand
 
    Next
 
END SUB

You can also use deferred addressing to refer to only part of a variable name (although it must be a complete branch). The previous example could be re-written as follows.

SUB Init ()
 
    DIM VarName As Str;
 
    DIM Index as Integer;
 
    
 
    For Index = 0; Index < 11; Index ++
 
        VarName = AddString("Tank", TOC(Index));
 
        '-------- Deferred variable referencing
 
        ?Varname.Level.SP = 0;  '? with a work variable combined with additional hard coded branch/leaf segments
 
    Next
 
END SUB

The follow example will not work as it attempts to use deferred addressing in a context other than the left-side operand of a value assignment.

'Prerequisite: The I/O or internal variable called Recipe.1.RecipeName has the value "hello".
 
Sub Main()
 
    Dim TxtContent as Str;
 
    Dim RecipeName as str;
 
 
 
    RecipeName="Recipe.1.RecipeName";
 
    Print(?RecipeName);            'Same result as calling Print(RecipeName). Will print the string "Recipe.1.RecipeName" as opposed to the value of the variable called Recipe.1.RecipeName
 
    TxtContent = ?RecipeName;    '? used as part of the right-side operand - Will fail
 
    Print(TxtContent);
 
End Sub
 
 
 
Results:
 
Recipe.1.RecipeName
 
hello
