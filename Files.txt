File Example 5 - Use of the ChildOf property
 

 
 
 
 
 
 
 
 
 
 
 
File Example 5 - Use of the ChildOf property
 
Example of XML file generation using the Application Architect using the File and File Item configuration elements.
 
This topic describes 2 usages of the ChildOf property to generate XML files where the XML hierarchy of elements does not match the topology of instances:
 
 
The first file shows how to generate a flattened XML file,
 
The second file shows how to group child elements together.
 
 
Desired output - File 1: The flattened list of buildings and floors
 

 

 
 
 

 
<?xml version="1.0"?>
 
 
 

 
<RootXmlFlat>
 
 
 

 
<FlatXml_ListOfBuildingsAndFloors>Building01</FlatXml_ListOfBuildingsAndFloors>
 
 
 

 
<FlatXml_ListOfBuildingsAndFloors>Building01.Floor01</FlatXml_ListOfBuildingsAndFloors>
 
 
 

 
<FlatXml_ListOfBuildingsAndFloors>Building01.Floor02</FlatXml_ListOfBuildingsAndFloors>
 
 
 

 
<FlatXml_ListOfBuildingsAndFloors>Building01.Floor03</FlatXml_ListOfBuildingsAndFloors>
 
 
 

 
<FlatXml_ListOfBuildingsAndFloors>Building02</FlatXml_ListOfBuildingsAndFloors>
 
 
 

 
<FlatXml_ListOfBuildingsAndFloors>Building02.Floor01</FlatXml_ListOfBuildingsAndFloors>
 
 
 

 
<FlatXml_ListOfBuildingsAndFloors>Building02.Floor02</FlatXml_ListOfBuildingsAndFloors>
 
 
 

 
</RootXmlFlat>
 
 
 
 
 
 
 
Desired output- File 2 - With 2 separate collections - Buildings and Floors
 

 

 
 
 

 
<?xml version="1.0"?>
 
 
 

 
<RootXmlGrouped>
 
 
 

 
<Buildings Count="2">
 
 
 

 
<FlatXml_CollectionsOfBuildingsAndFloors>Building01</FlatXml_CollectionsOfBuildingsAndFloors>
 
 
 

 
<FlatXml_CollectionsOfBuildingsAndFloors>Building02</FlatXml_CollectionsOfBuildingsAndFloors>
 
 
 

 
</Buildings>
 
 
 

 
<Floors Count="5">
 
 
 

 
<FlatXml_CollectionsOfBuildingsAndFloors>Building01.Floor01</FlatXml_CollectionsOfBuildingsAndFloors>
 
 
 

 
<FlatXml_CollectionsOfBuildingsAndFloors>Building01.Floor02</FlatXml_CollectionsOfBuildingsAndFloors>
 
 
 

 
<FlatXml_CollectionsOfBuildingsAndFloors>Building01.Floor03</FlatXml_CollectionsOfBuildingsAndFloors>
 
 
 

 
<FlatXml_CollectionsOfBuildingsAndFloors>Building02.Floor01</FlatXml_CollectionsOfBuildingsAndFloors>
 
 
 

 
<FlatXml_CollectionsOfBuildingsAndFloors>Building02.Floor02</FlatXml_CollectionsOfBuildingsAndFloors>
 
 
 

 
</Floors>
 
 
 

 
</RootXmlFlat>
 
 
 
 
 
 
 
Topology
 

 
 
 
Project template
 

 
 
 
The Project template includes 2 files:
 
 
For file 1: FlatXml_ListOfBuildingsAndFloors
 
For file 2: FlatXml_CollectionsOfBuildingsAndFloors. 2 File items are defined, so that 2 Xml elements are generated in the output file. One is for the XML element corresponding to the collection of Buildings, the other for the XML element corresponding to the collection of Floors. You can notice the CountAttribute file item to generate the XML attribute for the element count in the respective collection.
 
 
File element on Project template for file 1 - FlatXml_ListOfBuildingsAndFloors:
 

 
 
 
Note the Tags to include and Root element, which will be referenced in File items.
 
File element on Project template for file 2 - FlatXml_CollectionsOfBuildingsAndFloors:
 

 
 
 
Note the Tags to include and Root element, which will be referenced in File items.
 
File items for the Buildings and Floors collections:
 

 
 
 

 
 
 
These 2 file items are direct children of the File itself. The topology hierarchy and the desired XML file hierarchy match, therefore the property ChildOf can be left empty.
 
Building and Floor templates
 

 
 
 
Both are designed in the same way, they include 2 file items:
 
 
For file 1: FlatXml_ListOfBuildingsAndFloors
 
For file 2: FlatXml_CollectionsOfBuildingsAndFloors.
 
 
File element on the Building and Floor templates for file 1: FlatXml_ListOfBuildingsAndFloors
 

 
 
 
Note the Tags to include and Root element. We reference the desired root XML element name in the ChildOf property to flatten the XML file.
 
File element on the Building template for file 2: FlatXml_CollectionsOfBuildingsAndFloors
 

 
 
 
Note the Tags to include and Root element. We reference the desired Buildings XML element name (root of the Buildings collection) in the ChildOf property to group buildings together in the XML file.
 
File element on the Floor template for file 2: FlatXml_CollectionsOfBuildingsAndFloors
 

 
 
 

 
Note the Tags to include and Root element. We reference the desired Floors XML element name (root of the Floors collection) in the ChildOf property to group floors together in the XML file. 
 
The same logic and design principles apply if you want to model and generate a file of type Predefined format for a tree-view. 
 
The only difference is that, where you reference the XML element name in the ChildOf property for an XML file, you will reference the name of a File element or the name of a File Item element in the case of the Predefined format.

Using files with templates
 

 
 
 
 
 
 
 
 
 
 
 
Using Files With Templates
 
Application developers often define and use files for specific purposes within the application. Typical examples include SCADA Basic and the population of form controls. The purpose of this feature is to allow the generation of such files from the Application Architect. 
 
Files can be generated in one of three formats.
 
 
Pre-defined - Text files with a format specifically for the form and pie/doughnut/pyramid controls.
 
XML - Text files with an XML compliant structure.
 
Text - Text files with free format.
 
 
To configure and generate the files two configuration elements are used.
 
 
File - Describing the file properties (name, format etc.) and an optional header and footer.
 
File item - Describing data to include in the file and its format.
 
 
All file, and file item, configuration properties support, wherever possible, the use of expressions and parameters. Several properties of both the file and file item are specific to the chosen file format, others are common. See the topics:
 
File and File Item common properties
 
 
File and File item properties specific to the pre-defined format
 
 
File and File item properties specific to the XML format
 
 
File and File item properties specific to the text format
 
 
To determine if a file item should be included in a file (or not) a tagging system is employed. See the topic Controlling instantiation using tagging. 
 
Changes to the Application Architect to assist with the support of file generation
 
The following changes were made, from version 12 of PcVue, to support file generation.
 
 
Elements are sorted alphabetically before generation. This has the effect of presenting generated elements in the same order as the display. 
 
The root of the application is now a topological element. It can be transformed into a template and therefore contain any configuration element including files and file items.

File and File Item Properties Specific to the Predefined Format
 
Predefined format
 
The following properties are for File and File Item objects when using the predefined format. This format is to support the PcVue form controls: combo box, list box, tree view.
 
The tree-view is the most complex because it can manage the tree structure with hierarchy and behaves in a similar manner to the XML format. 
 

 
XML is a hierarchical format but it should not be confused with the hierarchy of the AA. While in the AA there is a direct hierarchical link between parent and child (for example Building and Floor), this does not imply a direct hierarchical link between an XML element positioned on Building and an XML element positioned on Floor.
 
 
File:
 
 
There are no file properties specific to the predefined format.
 
File item:
 
 
 
 
 
 
 

 Property
 Description
 Default value or example

 The value of the item's UserData field.
 Example: =Me.Name

 The item's Text field in language 1.
 Example: =Me.DescriptionLang0

 The item's Text field in language 2.
 Example: =Me.DescriptionLang1

 Set to 1 if this item must be selected by default when opening the control.
 Default: 0
 
 

 Collapsed icon type
 The type of icon to display when the node is collapsed.
 &#160;
 
 

 Expanded icon type
 The type of icon to display when the node is expanded.
 &#160;
 
 

 
 
Child of
 
 The name of the parent element. If empty, it will be the last hierarchically.

Animations and the tree view control:
 
 
Only the following animations are currently supported by the file item when used with a tree view.
 
 
Link open
 
Send program
 
Send bit
 
Send register
 
Send text
 
Send macro
 
Recipe
 
 
The animation is created as child of a file item object. If multiple animations are added to the same object, they are processed in alphabetical order. Any animations added manually using the PcVue mimic editor will be lost at the next generation.
 
Notes on using the ChildOf property
 
The ChildOf property makes it possible to have grouping that does not depend on the topological hierarchy. If using this property, it is strongly advised that all applicable elements use it. See File Example 5 - Use of the ChildOf property for examples specific to the use of the ChildOf property.
