Installation and configuration
 

 
 
 
 
 
 
 
 
 
 
 
Installation and Configuration
 
For PcVue to work with ICCP features, the application designer must:
 
 
Have a license with ICCP Server option enabled. The driver can run in demo mode for one hour. However, any networking architectures do not work in demo mode.
 
Reference the svmgrICCP.dll file in the usrmgr.dat file that is in the Bin folder.
 
Have a configuration file, ICCP.ini, which must be located in the C folder of the project. It is used to configure the ICCP LCCs in PcVue.
 
 
The usrmgr.dat file should have the following content:
 

 

 
 
 

 
[USRMGR\svmgrICCP]
 
 
 

 
DLL=svmgrICCP.dll
 
 
 
 
 
 
 
ICCP.ini file
 
Some fields of the ICCP.ini file are mandatory. If they are missing or invalid, the driver fails to start. Other fields are optional and if they are missing or invalid, the task uses default values. The Event Viewer displays warning messages in the event of a missing or invalid field.
 
To comment a line in the ICCP.ini file, you can enter a single quotation mark (�) at the beginning of the line.
 
The following paragraphs describe the ICCP.ini file in more details.
 

 
Name fields are used as unique identifiers for the configured items. Their value must be of type String with a max length of 20 characters.
Allowed characters are: [a-z], [A-Z], [0-9] and underscore.

Application References sections 
 
The sections [ARn] in the ICCP.ini file define the addressing information necessary to define LCCs or RCCs. Depending on the use of the AR, the fields are associated to either local or remote addresses.
 
There are different functions to the application references depending on the Local Control Center type (Calling Client/Server, Called Client/Server, etc). Also, most of the fields in the ARs are optional, which can have different meanings:
 
 
If it is a Called type, omitting a specific field implies that upon remote connection, the local control center does not require an exact match of that field to accept or deny the connection.
 
If it is a Calling type, omitting a specific field implies that when the local control center sends the connection request to the remote, it does not include that field.
 
 
The exact requirements of each connection to a remote is handled between the two parties.
 

 
The TSelector field is particularly important. To avoid connection issues, it is recommended to use different TSelectors for different LCCs.
 
To differentiate application references, the user must create different blocks in the ICCP.ini file in a section called [ARX] where X is an increment starting from 1.

Networks sections
 
The sections [NetworkX] in the ICCP.ini file define a Network that contains any number of Local Control Center objects. The Network is mainly useful to separate different PcVue stations and behaviors. To differentiate networks, the user must create different sections in the ICCP.ini file, each named [NetworkX] where X starts at 1.

 Value
 Description

 <Name>

Determines the name of the Network.
 

 
We recommend to use a short name. A long name will affect variable creation, which is explained in following sections.
 
 
If it is empty, the Network is not created.

LCCs sections
 
The sections [NetworkX/LocalControlCenterY] in the ICCP.ini file define the information of a local control center to be created. Multiple local control center can be created.
 
If multiple LCCs are configured, it is important to pay close attention to:
 
 
The AR, in particular the IP Address and the Port that must be different for the different configured LCCs when not using Full 7 Layer Stack addressing.
 
All the AR fields when using Full 7 Layer Stack addressing. All the LCCs can share the same port in this case.
 
 

 
The TSelector field in the ARs is especially important to differentiate the different LCCs that are being created.
 
 
All LCCs must be configured linked to a Network. To differentiate LCCs, the user must configure a different section for each LCC (in the ICCP.ini file) called [NetworkX/LocalControlCenterY] where X and Y increase starting from 1.

The Quality Current Source flag specified in the variables extended attribute if the variable is not simulated.

RCCs sections
 
The sections [NetworkX/LocalControlCenterY/RemoteControlCenterZ] in the ICCP.ini file define the information of an RCC to be created. Multiple RCCs can be created for each local control center that is defined.
 
Remote Domains are included in the fields of this section. The VCC remote domain is automatically create for each RCC.
 
All RCCs must be created linked to a Local Control Center. To differentiate RCCs, the user must create different sections in the ICCP.ini file called [NetworkX/LocalControlCenterY/RemoteControlCenterZ] where X, Y and Z increase starting from 1.

Configuring ICCP variables
 

 
 
 
 
 
 
 
 
 
 
 
Configuring ICCP Variables
 
The user can create and configure ICCP variables in PcVue by adding a formatted value in the extended attributes of the variable. This value contains ICCP mapping information for that variable. It has 7 or 9 fields, depending on the type, with each field separated by a special character corresponding to the parameter BranchSeparator.
 
The user can configure two important parameters in the ICCP.ini file that are later used to parse the variables extended attributes:
 
 
BranchSeparator - The field separator in text extended attributes
 
AttributeNumber - The Id of the text extended attribute that contains the ICCP mapping
 
 

 
The number of characters in an extended attribute is limited to 100. To reduce the length, some of the more common fields are entered with a single or a few characters.
 
 

 
It is important that to use short names with care for the elements Network, LCC, Local Domain, RCC, Remote Domain and Data Set. All the name lengths come into play when using this extended attribute text.
 
 
Required parameters
 
In the extended attribute, the user must enter specific information that is important to define the variable mapping. All the fields must be separated by the character BranchSeparator defined in the ICCP.ini file. The specific order of these fields depends on the variables type (data or device command) and its origin (local or remote).

A header that must be ICCP. Mandatory for all ICCP variables. 
 
Used to identify that this is an ICCP variable.
 
Case sensitive.

More information about <MappingId>
 
 
For ICCP States, you can select to use a bit, register or alarm variable.
 
All Discrete, Real, ControlCommand, DiscreteSetPoint and RealSetpoint variables use F as their mapping ID. ICCP State variables have 4 possible states: 0, 1, 2 and 3. Typically, 0 is intermediate, 1 is open, 2 is closed and 3 is error.
 
A register mapped to a state, functions in the following way:
 
As a register it can only have F as its mapping Id.
The value 0 represents a State value of 0 (00 or intermediate).
The value 1 represents a State value of 1 (01 or open).
The value 2 represents a State value of 2 (10 or closed).
Any other value represents a State value of 3 (11 or error).
 
A bit or alarm can function in the same way.
 
If the mapping Id is F and the bit is valid in PcVue, it can send only 2 values. This option can be inverted when used in conjunction with the InvertStateSingleBit parameter discussed previously.
The value of 0 represents a State value of 1 (01 or open).
The value of 1 represents a State value of 2 (10 or closed).
If the bit variable is not valid (Status NS) it represents a State value of 3 (11 or error).
 
If the mapping Id is not F, then the ICCP State must be represented with 2 bits or 2 alarms. One of the bits will represent the High part of the bitstring and the other bit will represent the Low part.
 
H and L must be used together. They must be associated to the same <ICCPVarName> but with different PcVue variable names.
Example: 2 PcVue bit variables with the following fields in the extended attribute:
SVBit1 -> ICCP/MyState/H/Dom/Discrete/Telemetered
SVBit2 -> ICCP/MyState/L/Dom/Discrete/Telemetered
When SVBit1 is 0 and SVBit2 is 0, they represent an ICCP state MyState of 0 (00 or intermediate)
When SVBit1 is 0 and SVBit2 is 1, they represent an ICCP state MyState of 1 (01 or open)
When SVBit1 is 1 and SVBit2 is 0, they represent an ICCP state MyState of 2 (10 or closed)
When SVBit1 is 1 and SVBit2 is 1, they represent an ICCP state MyState of 3 (11 or error)
In this High/Low configuration of the states, the timestamp that is sent to the ICCP variable comes directly from the variable�s origin.

Overview of the ICCP Add-on
 

 
 
 
 
 
 
 
 
 
 
 
Overview of the ICCP Add-on
 
ICCP is a protocol, also known as ICCP/TASE.2, standardized as part of the IEC 60870-6 series of standards. It operates over TCP/IP to connect control centers in a power system. A control center identifies itself on the network with a series of parameters (IP Address, App Title, etc.). ICCP can connect a server to one or several clients as required. Data frames are delivered using MMS (Manufacturing Message Specification) messages. 
 
ICCP uses two mechanisms to transfer data: User requests and Object changes (exceptions).
There are two classes of data objects:
 
 
Data value for monitoring,
 
Device command value for commands.
 
 
Abbreviations
 
The following abbreviations and acronyms may be found in this help book:
 
 
AE -	Application Entity
 
AP - Application Process
 
AR - Application Reference
 
BT - Bilateral Table
 
CBB - Conformance Building Block
 
DE - Direct Execute
 
DS - Data Set
 
DSTS - Data Set Transfer Set
 
ICC - Inter-Control Center 
 
ICCP - Inter-Control Center Protocol 
 
IEC - International Electrotechnical Commission 
 
IP - Internet Protocol
 
LCC - Local Control Center 
 
MMS - Manufacturing Messaging Specification
 
NS - Not Set
 
PDU - Protocol Data Unit
 
RCC - Remote Control Center
 
SBO - Select Before Operate
 
TASE - Telecontrol Application Service Element, IEC�s designation of an international standard protocol for utility data exchanges
 
TASE.2 - TASE version based on the ICCP protocol
 
TCP - Transmission Control Protocol
 
VCC - Virtual Control Center
 
VMD - Virtual Manufacturing Device
 
 
VCC vs ICC Domains
 
There are two types of domains in ICCP. The first one is the VCC, which is a global scope domain. The second is an ICC and it is a user-generated domain. The VCC scope is higher than an ICC scope. Domains are mainly used to group variables up.
 
A remote client can access a local server's ICC domain and the variables the server publishes and still have access to any variables published in the VCC domain. However, the local server must have enough DSTransfer Sets available to share with all the remote clients that connect. If a remote client attempts to connect and the local server does not have an available DS Transfer Set, the remote will not be able to receive advanced data transmission schemes.
 
The remote client requires the existence of a remote domain (a local domain in the PcVue application�s perspective) in order to be able to connect. Because of this, a local Server or Client/Server has to create at least one Local Domain (ICC), even if the remote only require access to VCC variables.
 
Basic ICCP model structure
 
To create a local ICCP server that can communicate with one or multiple clients the application designer must configure the following settings as a minimum.
 
 
The 7 layer stack settings for the local server to be created,
 
An ICC domain and bilateral table information,
 
Local variables belonging to either the ICC or the VCC for the server to publish.
 
 
More specific information can also be entered, however this is optional. The user can enter these ICCP parameters to be more specific when creating the Local Control Center. The optional parameters are explained in more detail in the following chapters.
 
For the local ICCP client, it is necessary to have at a minimum:
 
 
The 7 layer stack information for the local client, which includes the 7 layer stack info of the remote,
 
One ICC domain defined,
 
One Data Set defined with its associated DS transfer set,
 
Remote variables belonging to either the ICC or the VCC for the client to subscribe to.
 
 
For the local server part, it is necessary to have at least one ICC domain and its associated bilateral table ID for the communication to work, even if the user does not want variables in the ICC Domain (and has all of the variables in the VCC).
 
For every local domain that is created, the server will automatically create 10 DS Transfer sets for the clients to access from the local server. This parameter can be changed if required with the parameter LocalServerMaxDSTS.
 
For the local client part, it is necessary to have the remote information that the client is connecting to, as well as the local addressing information.
 
When creating a bidirectional exchange, it is necessary to define the parameter that indicate which control center is going to initiate the communication.
 
Structure
 
In PcVue, the ICCP model consists of different objects that are nested in the following way:

ICCP AR
ICCP Network
 
ICCP Local Control Centers
 
ICCP Local Domains
 
ICCP DSTS
ICCP Remote Control Centers
 
ICCP Remote Domains
 
ICCP DS

ICCP redundancy
 

 
 
 
 
 
 
 
 
 
 
 
ICCP Redundancy
 
The ICCP Add-on supports 2 redundancy mechanisms that can be combined:
 
 
Redundancy of ICCP interfaces when more than one Application References are configured for a given remote control center,
 
Redundancy as part of the networking system when more than one PcVue station support as an ICCP interface.
 
 
Redundancy of ICCP interface for remote control centers
 
When an ICCP control center has the Calling behavior, or when it initiates communication (Client only or Client/Server that initiates communication):
 
 
When a remote control center fails or disconnects for any reason, PcVue ICCP local control center goes through a list of backup remote control centers, defined in the ICCP.ini file, to reestablish connection.
 
The backup remote control centers are only differentiated by their Application Reference and these are entered in the RemoteARX field of the Remote Control Center, where X can be from 1-10 .
 
If none are available, the Local Control Center will attempt to reconnect to the defined remote control centers in order, after the InitiateRetryDelay timer of the local control center passes.
 
If the entire first cycle passes and no backup center is available for connection, PcVue will set the status of all remote variables to NS (invalid).
 
 
When an ICCP control center has the Called behavior, or when it does not initiate communication (Server only or Client/Server that does not initiate):
 
 
The behavior of remote disconnections works by waiting a given number of seconds (as defined by the parameter CalledDisconnectTimeout) to let a remote control center reconnect. If this timeout expires, then the remote data variables are set to quality NS (invalid).
 
 

 &#160;PcVue Networking
 
When creating a redundant PcVue architecture, the ICCP variables are handled based on the producer/consumer principle. Variables on the active station are published to all the passive stations or client stations. Any modification to a variable on any of the networking stations are detected and updated by the ICCP driver.
 
For such high availability architectures to operate properly, the ICCP.ini file must be configured as follow:
 
 
The PcVue association name must be included in the field AssociationName of the Network.
 
The IPAddress field is most likely different for each station that is expected to maintain ICCP communication. The active station has one IP address, and the passive station has a different one. It is important to pay attention to this detail for proper operations. 
It is possible to leave the IPAddress field empty, and the driver uses the localhost address.
 
 
When there is a switchover between active and passive stations:
 
 
Calling Client/Servers: The newly active station will cycle across its remote control centers, trying to connect to one of them before setting remote data variables to status NS (invalid).
 
Called Client/Servers: The newly active station will wait a given number of seconds (as defined by the parameter CalledDisconnectTimeout) to let a remote control center reconnect. If this timeout expires, then the remote data variables are set to status NS (invalid).

ICCP log messages and traces
 

 
 
 
 
 
 
 
 
 
 
 
ICCP Log Messages and Traces
 
There are two types of traces that will appear in the Event Viewer of PcVue, and in the Log Files: Log Messages and Traces.
 
Log Messages
 
This type of message is used for events that occur when the project starts (ICCP Control Center object creation and variable creation/modification) and with different connection events (connect, disconnect, etc.). This type of message can be configured in the ICCP.ini file with the field LogMessageLevel. There is a hierarchy of Log Message Levels for the different types of ICCP objects (Global/Family, Network, Local Control Center and Remote Control Center). Each one has its own LogMessageLevel, and if Default is selected, the specific object will use the LogMessageLevel of its parent object. During runtime it is possible to modify the LogMessageLevel of each object with the SCADA Basic verbs. However, the changes done at runtime will only persist while the project is running, they will not be saved to the ICCP.ini file.
 
It has five different level of details that the user can select for their purposes. A higher level includes the messages of all lower levels
 
 
LogMessageLevel = 0: Default � This level is the default option for all ICCP objects (except for global). Using this level will apply the Log Message Level of the parent object.
 
LogMessageLevel = 1: Minimum � This level will show the most basic messages that include detected errors in the inputs of the ICCP.ini file and in the variable creation. It will also display a basic message for connections/disconnections of remote control centers.
 
LogMessageLevel = 2: Light details � This level includes more details of the control center creation, variable creation and connections/disconnections of a remote control center.
 
LogMessageLevel = 3: Full details � This level has the most level of detail of the different events while still maintaining a tracing structure.
 
LogMessageLevel = 4: Stack � This level includes messages directly from the underlying ICCP driver kernel. This implies that there will be less order in the messages.
 
 
It is important to note that Log Messages are not designed to display events that occur cyclically, like variable updates.
 
Traces
 
This type of messages are used for events that occur cyclically and must only be activated temporarily for diagnostic purpose. To enable these traces, the user must create a SCADA Basic program inside the PcVue project that enables the traces. The code required is the following:
 
To enable:
 

 

 
 
 

 
TRACE(1,&#160;11,&#160;"03");
 
 
 
 
 
 
 
To disable:
 

 

 
 
 

 
TRACE(0,&#160;11,&#160;"03");
 
 
 
 
 
 
 
 
The first argument can be 1 (to enable) or 0 (tor disable).
 
The second argument is the dll number that must be 11 if the ICCP dll is the first SV Manager Add-on currently loaded.
 
The third argument is an hexadecimal number that represents the bit mask that are selected for the trace to be enabled or disabled.

Timestamps
 
The origin of the timestamp that is sent to the remote control center via ICCP can be:
 
 
Internal: If this is the case (for internal variables or I/O variable not timestamped at the source), then the timestamp that is used is the one from PcVue (SV status).
 
External: If the variable comes from a device supporting timestamping at the source (IE C104, DNP3, etc.) then the timestamp that is used is the one from the device (PLC status).
 
 
The timestamps that are received from the remote control center via ICCP can be:
 
 
Valid: It is marked by Good Quality in PcVue and represented by the status PLC.
 
Not Valid: It is marked by Bad Quality in PcVue. It does not have the status PLC.
 
 
The precision of the timestamp in ICCP depends on the variable type:
 
 
Second precision for some data types.
 
Millisecond precision for others.
 
 
Change of value counter (COV)
 
The COV counter is only available for DiscreteExtended, RealExtended and StateExtended ICCP variable types. This counter increases every time the value is updated. In the current version, the COV counter is not mapped.
 
 
For local server variables: The COV is updated and sent to the remote. The value can be seen in the traces for diagnostics.
 
For remote server variables: The COV is received from the remote. The value can be seen in the traces for diagnostics.
