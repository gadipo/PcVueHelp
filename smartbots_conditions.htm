Conditions

The bot supports the following types of conditions:

  • Local database changed

  • Expression

  • User information changed

  • Schedule

  • Memory

  • Message

  • Register variable

  • Text variable

  • Bit variable

  • Delay

  • Webhooks

  • Event changed

To create a new condition, first select the transition or trigger to which the condition will be added. Then right click to open the context menu, or use the New condition button in the tool bar. Select the required operation from the drop-down menu.

Local database changed

Monitors changes in the specified local database property. Note that the monitoring is starting only when the smart bot’s process starts to verify the condition. Thus, if the condition is in the trigger, then all database changes are captured. If the condition is in the transition, smart bot will verify only those changes that happened after entering the corresponding state.

If the condition is in the trigger, corresponding database changes are monitored and collected in a queue for one hour. Thus if the smart bot detected an appropriate change and started to handle it, and meanwhile the second change has happened, it will be buffered and stored until the smart bot re-verifies the condition again or one hour passes.

This condition has the following properties.

Property name Type Description
Description String Displayed name of the condition. This property is only used in the HMI and has no effect at run-time.
Database name String Name of the database table. This property supports usage of internal bot variables.
Change types Enum

Type of database changes to monitor. The following values are available.

  • Any change

  • Insert

  • Update

  • Delete
Condition C# lambda expression

C# lambda expression – function taking the database rows as an input and filtering which rows to verify. This property supports usage of internal bot variables.

As an example, the following string can be used to monitor all rows having the value of ABC property less than 30. x => x.ABC < 30

The following string can be used to monitor all rows having the value of ABCD property equals to "test". x => x["ABCD"] == "test"

User information changed

Monitors changes in the specified user property. Condition can be satisfied only on the second read of the user information or later. The first read initializes the last known value, then subsequent reads are matched with the last known value. This condition has the following properties.

Property name Type Description
Description String Displayed name of the condition. This property is only used in the HMI and has no effect at run-time.
User name String

Name of the user whose information needs to be monitored. This property supports usage of internal bot variables.
If value is left empty, all users will be monitored. Once the corresponding property is changed for one of them, the user name, the previous value and the new value are set as the current context and can be received by the Save to memory operation.
Information to retrieve Enum

The property of a given user to monitor. The following values are available.

  • Display name

  • Description

  • E-mail

  • Phone number

  • Title

  • First name

  • Last name

  • Profile level

  • Permission level – access level of the user in the current chat

  • Occupation

  • Availability – free/busy status of the user

Runtime property initialized by the scheduler. Note that connected MS Graph plug-in is required to receive this property.

  • Mobile connectivity – specifies if the user is currently connected via Mobile application

  • Latitude

  • Longitude

  • Name of the current zone

  • Description of the current zone

The tracking feature is required to receive the position and location related properties.

Message

Verifies the last message in the given chat. Condition will be satisfied when all the condition properties match the corresponding properties of the last message in the Chat. This condition has the following properties.

Property name Type Description
Description String Displayed name of the condition. This property is only used in the HMI and has no effect at run-time.
Display name String Sender of the message. This property supports usage of internal bot variables.
Message type String Type of the message or extension of the attachment. This property supports usage of internal bot variables.
Media type

Text, Audio, Photo, Video, Document

 Media type of the message. This property is used to specify which kind of messages is expected. E.g. Photo media type corresponds to all message types related to the Photo attachment.

Webhook

A Webhook condition allows bots to catch and handle external webhooks.

The Webhook format is not standard, with each 3rd party system using webhooks defining their own API and payload for the webhook. Therefore there is a set of supported webhook definitions and types. When calling a webhook, a 3rd party system must specify a security code in the Webhook request. The Messaging Web Service ignores all requests without a code or with a wrong code. The correct code is defined in the Messaging Web Service configuration file. Show picture

When catching a webhook, the Bot saves its payload to a context that can be extracted using the SaveToMemory operation.

The Webhook condition has the following properties.

Property name Type Description

Description

String

Displayed name of the condition. This property is only used in the HMI and has no effect at run-time.

Type

String

The type of the webhook to catch. The following values are available.

  • Custom JSON webhook

  • Generic JSON webhook

  • Azure Alert

  • Kudu notification - Stripe

Condition

C# lambda expression

Function taking the webhook payload as an input and filtering which webhooks to catch. This property supports usage of internal bot variables.

  • Custom JSON webhook - For Custom JSON webhooks, the Messaging Web Service accepts ordinary get/post HTTP(s) requests on the following endpoint.
    https://<host>/Messaging/v2/Webhook?code=<securitycode>&<other query arguments>
    For GET methods query parameters are used as webhook payload. For POST methods the body is expected to be in JSON format which is used as the webhook payload.

  • Generic JSON webhook - For Generic JSON webhooks the Messaging Web Service accepts post HTTP(s) requests on the following endpoint.
    https://<host>/api/webhooks/incoming/genericJson

  • Azure Alert - For Azure Alert webhooks the Messaging Web Service accepts post HTTP(s) requests on the following endpoint.
    https://<host>/api/webhooks/incoming/azurealert

  • Kudu notification - For Kudu notifications the Messaging Web Service accepts post HTTP(s) requests on the following endpoint.
    https://<host>/api/webhooks/incoming/kudu

  • Stripe - For Stripe webhooks the Messaging Web Service accepts Post HTTP(s) requests on the following endpoint.
    https://<host>/api/webhooks/incoming/stripe
