ICCP Redundancy

The ICCP Add-on supports 2 redundancy mechanisms that can be combined:

  • Redundancy of ICCP interfaces when more than one Application References are configured for a given remote control center,
  • Redundancy as part of the networking system when more than one PcVue station support as an ICCP interface.

Redundancy of ICCP interface for remote control centers

When an ICCP control center has the Calling behavior, or when it initiates communication (Client only or Client/Server that initiates communication):

  • When a remote control center fails or disconnects for any reason, PcVue ICCP local control center goes through a list of backup remote control centers, defined in the ICCP.ini file, to reestablish connection.
  • The backup remote control centers are only differentiated by their Application Reference and these are entered in the RemoteARX field of the Remote Control Center, where X can be from 1-10 .
  • If none are available, the Local Control Center will attempt to reconnect to the defined remote control centers in order, after the InitiateRetryDelay timer of the local control center passes.
  • If the entire first cycle passes and no backup center is available for connection, PcVue will set the status of all remote variables to NS (invalid).

When an ICCP control center has the Called behavior, or when it does not initiate communication (Server only or Client/Server that does not initiate):

  • The behavior of remote disconnections works by waiting a given number of seconds (as defined by the parameter CalledDisconnectTimeout) to let a remote control center reconnect. If this timeout expires, then the remote data variables are set to quality NS (invalid).

 PcVue Networking

When creating a redundant PcVue architecture, the ICCP variables are handled based on the producer/consumer principle. Variables on the active station are published to all the passive stations or client stations. Any modification to a variable on any of the networking stations are detected and updated by the ICCP driver.

For such high availability architectures to operate properly, the ICCP.ini file must be configured as follow:

  1. The PcVue association name must be included in the field AssociationName of the Network.
  2. The IPAddress field is most likely different for each station that is expected to maintain ICCP communication. The active station has one IP address, and the passive station has a different one. It is important to pay attention to this detail for proper operations.
    It is possible to leave the IPAddress field empty, and the driver uses the localhost address.

When there is a switchover between active and passive stations:

  • Calling Client/Servers: The newly active station will cycle across its remote control centers, trying to connect to one of them before setting remote data variables to status NS (invalid).
  • Called Client/Servers: The newly active station will wait a given number of seconds (as defined by the parameter CalledDisconnectTimeout) to let a remote control center reconnect. If this timeout expires, then the remote data variables are set to status NS (invalid).
