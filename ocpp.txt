How to install and configure the OCPP driver
 

 
 
 
 
 
 
 
 
 
 
 
How to install and configure the OCPP driver
 
The OCPP driver is an Add-on implemented based on the SV Manager SDK. It is made up of:
 
 

 
A dll named svmgrOCPP.dll installed in the \Bin folder as part of the PcVue installation.
 
 

 
A shared library named SH_EV_Charging installed as part of the installation. It includes ready-to-use templates you can instantiate with the Application Architect.
 
 

 
Dependencies of the driver, not provided as part of the installation and to be copied in the folder \Bin\Modules\OCPP.
 
 
 

 
You should contact your local reseller to get a copy of the package containing the dependencies to be copied in the folder \Bin\Modules\OCPP.
 
 
Prerequisites:
 
 

 
You have a license with the OCPP option enabled. The driver can run in demo mode for 1 hour with 1 charging point.
 
 

 
You have the OCPP driver binary files, including the OCPP module dependencies,
 
 

 
You have the shared library with the templates for OCPP.
 
 
 
Installing and activating the OCPP driver
 
 

 
If PcVue is running, make sure you shut it down.
 
 

 
Check that the driver's main dll named svmgrOCPP.dll is already installed in the \Bin folder.
 
 

 
Open the usrmgr.dat file in the PcVue�s Bin folder, add the following content and save the modified file:
 

 

 
 
 

 
[USRMGR\svmgrOcpp]&#160;
 
 
 

 
DLL=svmgrOcpp.dll
 
 
 
 
 
 
 
 

 
Extract the driver's dependencies obtain from your your local reseller int the \Bin\Modules\OCPP\ folder.
 
 

 
Configure the endpoint by performing the Url namespace reservation as described in the next paragraph.
 
 
 

 
If the usrmgr.dat file is missing, you can simply create it and place the above content in it.
 
 
Configuring the endpoint
 
The OCPP driver is implemented based on the Microsoft Windows http.sys web server. Before the OCPP driver can receive and process requests from charge points, http.sys must be configured to dispatch them properly based on the Url namespace reservation mechanism. The reservation gives listed users the right to run a process that listen on that part of the namespace and should be performed for each Http endpoint configured in the OCPPBinding section of the .ini file.
 

 
Reserving the Url namespace is mandatory so that PcVue can listen to incoming requests when running without administrative privileges.
 
 
Open a Windows command line or a PowerShell terminal and type the following command:
 

 

 
 
 

 
netsh http add urlacl url=http://[HOSTNAME][PORT]/ user=[USER ACCOUNT]
 
 
 
 
 
 
 
 Where:
 
 

 
http://[HOSTNAME][PORT]/ is the Url of the PcVue host charge points connected to.
 
 

 
[USER ACCOUNT] is the name of the user or group of users running the main process of PcVue (Sv32.exe).
 
 
 
Examples:
 

 

 
 
 

 
netsh http add urlacl url=https://my-csms.local:8080/ user=Domain\Sv_Users
 
 
 

 
netsh http add urlacl url=https://+:8080/ user=Sv_User
 
 
 
 
 
 
 

 
The firewall must also be configured to allow incoming Http requests.
 
 

 
When creating a WebSocket connection to a 3rd party system PcVue sends an HTTP header of type "Connection�. The value of this header consists of two directives: �Upgrade,Keep-Alive�. However, when evaluating the HTTP response to this message, PcVue is unable to parse a value that contains more than one directive. In other words, the value of the HTTP header �Connection� in the HTTP response must be nothing else than �Upgrade�.
 
 
Configuring the OCPP driver
 
The OCPP driver is configured using a .ini file named ocppConfiguration.ini. It includes:
 
 

 
General settings.
 
 

 
OCPP bindings configuration.
 
 

 
Charge points configuration.
 
 

 
Proxy configuration.
 
 
 
Some fields of the ocppConfiguration.ini file are mandatory. If they are missing or invalid, the driver will fail to start. Other fields are optional and if they are missing or invalid, the task will use default values. The Event Viewer displays warning messages if a field is missing or invalid.
 
To comment a line in the .ini file, you can enter a single quotation mark (�) at the beginning of the line.
 

 
If you plan to use the Application Architect and the predefined templates provided with the OCPP library, the ocppConfiguration.ini file will be generated automatically. Refer to the topic How to use the OCPP library for more information.
 
 
 

 
Go to your PcVue project's C directory and open the ocppConfiguration.ini file. Create the file if it does not exist. This file has several sections explained in the following table:
 
 
 
 
 

 Section
 Description
 
 

 General
 Driver's general settings.
 
 

 OcppBinding
 The TCP binding on which the OCPP server shall listen. Multiple bindings are possible, for instance http and https, or bindings for accessing the server from inside and outside of a public/private network boundary.
 
 

 Charge point
 Settings specific to an individual charge point. All charge points from which connections are accepted must be included under as many sections in the configuration file. The name of the section in the .ini file is the identifier of the charge point and must be unique, for example [CP1602]. The Id used as the section name shall match the identifier or name configured in the charge point itself (see the charge point vendor documentation for more information).
 
 

 Proxy
 Settings specific to proxying OCPP messages to a third-party central system. The name of the section is [Proxy:Name] where Name is used to identify the third-party central system and must be unique.
 
 
 
 
 

 
Define the names for all well-known charge points, each in a dedicated section and add the necessary keys to each section.
 

 
If a section is defined for a charge point which incoming requests should be ignored, because it is not yet in service for example, we recommend setting at minimum the Disabled key.
For example:

How to use the OCPP library
 

 
 
 
 
 
 
 
 
 
 
 
 
 
How to use the OCPP library
 
Charge point variables are the carriers of information exchanged between charge points on the network and PcVue. A library of predefined templates is available to ease the mapping of OCPP charging stations data and management messages.
 
The predefined library for OCPP includes:
 
 

 
Images
 
 

 
SCADA Basic scripts
 
 

 
Animated symbols
 
 

 
Templates and Parameters
 
 

 
Mimics
 
 

 
An archive unit
 
 
 
Prerequisites: 
 
 

 
You have installed and activated the OCCP driver, and performed Url namespace reservation as detailed in the topic How to install and configure the OCPP driver.
 
 

 
You have the OCPP shared library installed (folder \LIB\SH_EV_Charging).
 
 

 
You know how to use the Application Architect to instantiate existing templates.
 
 
 
Instantiating OCPP templates
 
 

 
Click Configure in the main menu of PcVue then Application Architect.
 
 

 
Click the plus button next to OCPP in the Templates tab. The categories OCPP 1.6 and 2.0 appear, containing templates for all OCPP variables. Show picture
 

 
 
 
 

 
Add an instance of the ChargePointProject template and give it a name. This instance provides:
 
 

 
Variables global to your charge point central monitoring system,
 
 

 
The essential keys for the General and OcppBinding sections of the OcppConfiguration.ini file,
 
 

 
The mimics global to your CSMS,
 
 
 
 

 
Set the value of input parameters for the essential keys in the .ini file, at least the Http endpoint parameter.
 
 

 
Add an instance of the ChargePoint template as a child of the ChargePointProject instance and give it a name that matches the Id of the actual charge point deployed on the field (for example CPxxxx). This instance provides:
 
 

 
Variables to monitor the charge point,
 
 

 
Inherited templates to support the related of OCPP messages: Status, authorization, handling of reservation, availability, charging profile, firmware update, etc.
 
 
 

 
Make sure that the name of the ChargePoint instance you add matches the name (or id) set in the corresponding charge point device. Refer to the charge point vendor documentation for more information.
 
 
 

 
Set the value of input parameters as required by your set-up.
 
 

 
For each connector on the charge point you have instantiated, add an instance of the Connector template as a child of the ChargePoint instance (index starts at 1). This instance provides variables and inherited templates to monitor the connector and support the related OCPP messages: Availability, electrical meter values, status, transaction, etc.
 
 

 
Set the value of input parameters as required by your set-up. At least the ConnectorId according to the charge point vendor documentation.
The phase number is also important if the charge point has more than one phase (possible values are: L1, L2, L3, L1L2, L2L3, L3L1, L1N, L2N, L3N, N).
 
 

 
Repeat steps 7 and 8 for each connector available on the charge point.
 
 

 
Repeat steps 5 to 9 for each charge point.
 
 

 
Save the configuration by clicking File.Save in upper menu then Tasks.Generate to generate the configuration.
 
 
Click Configure.Application Explorer in upper menu of PcVue then open the charge point project from Variables tree. All charge point messages and variables generated are displayed. Pre-animated mimics are also generated. Show picture

 

 
An alarm is set by the OCPP driver and is activated when the charge point status is Faulted and deactivated when it is any other value. Alarm and bit variable values are defined in relation to the values set in the associated labels of the Status variable.
The OCPP driver also writes label string representations in extended attributes of alarm and bit variables such as ChargePointErrorCode written into extended attribute 3.
 
We recommend you include additional details about charge point errors in other extended attributes.

OCPP Add-on overview
 

 
 
 
 
 
 
 
 
 
 
 
 
 
OCPP Add-on overview
 
The Open Charge Point Protocol (OCPP) is an interface for communication between Electric vehicle charging stations and a central management system. It was originally designed and is supported by the Open Charge Alliance (OCA).
 
The following OCPP versions are supported:
 
 

 
OCPP protocol revision 1.6 - OCPP via Json over WebSocket
 
 

 
OCPP protocol revision 2.0.1
 
 
 
&#160;PcVue acts as central management system to monitor charging stations. The OCPP specifications allows only one central system to exchange data with a charge point. In practice however, multiple stakeholders have an interest in the data obtained from a charge point. Therefore, PcVue supports the OCPP message proxying to handle the transfer of certain messages to a third-party central system. Show picture
 

 
 
 
OCPP messages are mapped onto PcVue variables to take advantage of the product capabilities:
 
 

 
Client/server distribution,
 
 

 
HMI display,
 
 

 
Alarming,
 
 

 
Archiving,
 
 

 
Subscription to data via server interfaces,
 
 

 
Etc.
 
 
 
There are no configuration dialog's specific to the OCPP Add-on. At the time of writing, it uses a dedicated .ini file and regular configuration features. In addition, a predefined library of templates is available to make it easy to configure charge points and variables using the instantiation capabilities of the Application Architect.
 
Glossary
 
The following terms and acronyms may be found in this help book:
 
 
OCPP - Open Charge Point Protocol.
 
EV - Electric Vehicle.
 
EVSE - Electric Vehicle Supply Equipment. An electric vehicle charging station, also called EV charging station, electric recharging point, charging point, charge point and electronic charging station, is an element in an infrastructure that supplies electric energy for the recharging of plug-in electric vehicles - including electric cars, neighborhood electric vehicles and plug-in hybrids.
 
Connector - The actual plug connected to the EV. A charge point may have one or several connectors that operate independently of each other.
 
CSMS - Charging station management system.
 
 

 
Refer to the topic OCPP protocol implementation statements for information on supported OCPP messages.

OCPP protocol implementation statement
 
OCPP specifies the exchange of messages between a charge point and a CSMS such as PcVue. OCPP messages are grouped in profiles. This topic includes detailed information about the level of support of OCPP by PcVue.
 

 
Only one CSMS can be connected with PcVue proxy at the time of writing.
 
 
Supported OCPP versions
 
 

 
OCPP protocol revision 1.6 - OCPP via Json over WebSocket.
 
 

 
OCPP protocol revision 2.0.1.

In other words, the value of the HTTP header �Connection� in the HTTP response must be nothing else than �Upgrade�.
