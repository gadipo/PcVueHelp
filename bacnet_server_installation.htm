Installation and Configuration

The BACnet Server Add-on is comprised of a single Dll file that must be copied to the Bin folder. The file is named:

  • svmgrBACnetServer.dll

The Add-on was developed using the SV Manager Toolkit. In order to load the Add-on at startup, a text file UsrMgr.dat must also be added to the Bin folder. The file must contain the following lines:

[USRMGR\BACnetServer]
 
DLL=svmgrBACnetServer.dll

In addition, the configuration of the Add-on is made-up of the following configuration files:

  • BACnetServerStack.ini - Defines the network interface and network-specific properties,
  • BACnetServerConfig.xml - Defines the BACnet Device object and services to expose,
  • BACnetServerUnits.xml - Defines the correspondence between the units used in PcVue and BACnet standard engineering units.

Network interface configuration file - BACnetServerStack.ini

The network interface and network-specific properties are configured using the file BACnetServerStack.ini that must be created in the C folder of the project.
A sample file, with default settings, is supplied with the Add-on and can be copied to the C folder of the project.

[general]
 
cnt_data_link = 1
 
max_msg_req = 100
 
 
 
[data_link 1]
 
port_id = 1
 
type = ip
 
lan_name = 0.0.0.0
 
net_number = 0
 
udp_port = 47808
 
max_msg_req = 50

To avoid potentials conflicts the lan_name should be set to be the same address of as the IP interface used by the server.

It is important to note that if the machine hosting the Server has more than one network card, the lan_name should match the IP address allocated to the network card that the Server is using.

BACnet Device configuration file - BACnetServerConfig.xml

An Xml file, BACnetServerConfig.xml, can be used to define optional configuration properties for the Server. If used, it must be located in the C folder of the project. 

<?xml version="1.0" encoding="utf-8"?>
 
<BACnetServerConfiguration>
 
    <BACnetObjectParametersExtendedAttribute>3</BACnetObjectParametersExtendedAttribute>
 
    <DeviceInstanceNumber>1990</DeviceInstanceNumber>
 
    <AutoCompletion>TRUE</AutoCompletion>
 
    <ClientsInformation>TRUE</ClientsInformation>
 
    <SegmentationSupport>TRUE</SegmentationSupport>
 
    <MaxADPULengthAccepted>1476</MaxADPULengthAccepted>
 
    <MaxNumberOfObjects>0</MaxNumberOfObjects>
 
    <DeviceRestartProcedure>1</DeviceRestartProcedure>
 
</BACnetServerConfiguration>
  • BACnetObjectParametersExtendedAttribute -The variable's Extended String Attribute number used to set the BACnet objects properties’ values (default - 3).
  • DeviceInstanceNumber - The exposed BACNet Device instance number (default - 1990).
  • AutoCompletion - Enable/Disable the Auto-Completion mechanism for the variable extended attribute definition (default - Enabled).
  • ClientsInformation - Enable/Disable the Services Notifications mechanism (default - Enabled).
  • SegmentationSupport - Segmentation support can be set to be supported in both directions (True) or in no case (False) (default - Supported in both directions).
  • MaxADPULengthAccepted - The Maximum ADPU frame length the device can accept (default - 1476).
  • MaxNumberOfObjects - The Maximum number of BACnet objects within the BACnet Server. If set to 0 then no limit is set (default - 0).
  • DeviceRestartProcedure - Enable/disable the Device restart procedure. If set to 1 the Device restart notification is broadcast upon start-up of the software. This notification is not sent if the parameter is set to 0.

Boolean values can be set using the following syntax:

  • 0, F, false, FALSE for value false
  • 1, T, true, TRUE for true

The configuration from this file is only loaded during the start-up of the project. Any change requires restarting the project.

If the file is not found during the project start-up, a default configuration will be loaded. If a property is missing or set with an incorrect value, the associated default value will be used. In both cases, diagnostic traces are generated.

An example is supplied with the Add-on, it can be copied to the C folder of the project and customized.

While this file is optional, it is important to have it properly configured in most cases, at least to adapt the DeviceInstanceNumber so that the Device Id exposed by the BACnet Server Add-on suits your need.

Engineering units configuration file - BACnetServerUnits.xml

To use the capability of exposing a register variable's Unit property as a valid BACnetEngineeringUnits, an Xml file called BACnetServerUnits.xml must be created in the C folder of the project.
This file must contain the correspondence between the strings used in the Unit property of variables and the standard BACnetEngineeringUnits numeric Id that shall be exposed.

A sample file An XML file, BACnetServerUnits.XML, containing correspondence between strings entered and the BACnetEngineeringUnits is also required to for the server. This file is supplied with the Add-on, and must be copied to the C folder of the project and customized.

Foreign device configuration - BACnetServerStack.ini

The BACnet Server Add-on can behave as a foreign device on a BACnet network.

If a BACnet Client is to interact with the BACnet Server Add-on as a Foreign Device, this BACnet Client must add the server to its Foreign Device Table, and the BACnet Server Add-on must register itself to the BBMD.

Configuration of Foreign Device handling is located in the BACnetServerStack.ini file.

[general]
 
cnt_data_link=1
 
max_msg_req=100
 
 
 
[data_link 1]
 
port_id=1
 
type=ip
 
lan_name=0.0.0.0
 
net_number=0
 
udp_port=47808
 
max_msg_req=50 
 
 
 
#Foreign device handling
 
foreign_dev_addr = 168.152.32.254:47808
 
foreign_dev_reg_timeout = 10
  • foreign_dev_addr - IP address and UDP port of the BBMD to register to,
  • foreign_dev_reg_timeout - Re-register time (in seconds).

The BACnet Server Add-on can behave as a foreign device on a BACnet network and register itself to a BBMD, but BBMD management itself is not supported.
