Smart Bots blueprint

When adding a new Smart Bot, or opening an existing one, the following page is displayed. It allows a user to define the bot states and transitions between these states, to define operations to execute in each state, and configure properties for all items. Show picture

The main panel represents all bot states and the transitions between them. The same objects are displayed in the tree view in the left panel. The properties of the selected item are displayed in the Properties Panel.

Tools

  • General tool panel - Allows a user to create a new Smart Bot, print the selected blueprint, open existing Smart Bots, save the current bot or save all opened bots.

  • Deployment tool panel - Tools to deploy the currently opened Smart Bot to the connected Instant Messaging Server, and to remove it from the server. These tools only work when the Instant Messaging Server is accessible, otherwise an error message appears.

  • Bot States tool panel - Each bot state is associated with a set of operations executed by the bot agent when entering the state. After executing the operations, the bot agent verifies all outgoing transitions to move to the next state. The Bot States tool panel contains tools to add a new bot state, add a new trigger, create a transition from the selected state, add an operation to the selected state and add conditions to the selected transition or trigger. The Enable all and Disable all tools allow a user to activate / deactivate all operations in the selected state and activate / deactivate all conditions in the selected operation, transition or trigger respectively. Every bot has an initial state.

  • Debugging tool panel - Tools to start and stop bot debugging. When debugging mode is activated, a special tool panel appears on the screen allowing a user to check the current state and values of variables for all deployed bot agents. The Debugging tool panel also contains tools to add a break-point, Continue the process, and move the process step by step.

  • Clipboard tool panel - Tools to cut, copy, paste and delete the selected object on the blueprint.

  • Organize tool panel - Allows a user to arrange bot states and transitions in a visually attractive way. The states and transitions display order has no effect at run-time.

Bot properties

Each bot has the following properties.

Property name Type Description
Bot name String Name of the bot, which can be referenced in the InviteUserToChat and RemoveUserFromChat operations. This name must be unique within a set of all deployed bots and all users in the project. If a user with the given name already exists in the project, the bot will not be loaded.
Bot description String Displayed name of the bot. This name appears in the chat.
Branch String Branch name applied to all PcVue variables in a bot’s operation and conditions.
Profile level [0...9] Level of the bot profile. If this level is less than the profile level of all chat participants, then the bot automatically gets access to the chat with ReadOnly rights.
Initial state State State in which the bot agent starts.
Bot type String

The type of bot selected from either Smart bot, Template, System or Assistant.

See the topic Smart Bot types for further information.

Bot states

Each bot state has a single property. This value is referenced in transitions and can be used as the Initial state of the bot.

Bot triggers

As with bot states, triggers have a Description property. This value is referenced in transitions and can be used as the Initial state of the bot.

A trigger contains a set of conditions that are verified on bot agent idle, that is after verifying all outgoing transitions from the current state. Note that, if there is no outgoing transition from the current state, this state is considered as final and the bot therefore proceeds to the completion behavior which is, by default, Return to the end of the last state. This completion behavior may be overridden after the bot entering the trigger state according to the corresponding parameter of the trigger.

The following completion behaviors are supported.

  • Stop procedure. Bot process stops and returns to a caller context if such exists.

  • Restart from the initial state. Bot restarts the procedure from the initial state keeping the existing memory.

  • Return to the last state. Bot returns to the last state before trigger has been fired.

  • Return to the end of the last state. Bot returns to the last state before the trigger has been fired, but not re-executing operations in that state.

The trigger conditions are verified before checking all other conditions of the current state. Thus if the state has a transition with a satisfied condition, the thread will not continue immediately to the following state, but verifies the triggers first. If a trigger also has a satisfied condition, then thread goes to the trigger branch.

When a bot enters a trigger branch, it stops verifying other trigger conditions until the trigger branch execution is completed. Giving priority to the execution of a trigger branch avoid uncontrollably switching between two trigger branches when both have satisfied conditions. A trigger branch execution is considered complete when the bot ends up in a final state (a state without any transition out).
