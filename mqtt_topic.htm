Using MQTT topics

Topics are at the heart of MQTT. They are a hierarchical tree-like structure, and MQTT subscribers subscribe to messages associated to a given topic.

For example, an MQTT system could be based on a topic structure such as BuildingId/FloorId/ZoneId/DataId, which would lead to handling messages associated to the following topics:

  • BuildingA/Floor7/ConferenceRoom/Temperature

  • BuildingA/Floor7/ConferenceRoom/Humidity

  • BuildingA/Floor7/ConferenceRoom/Light

MQTT includes a variety of mechanisms for handling topics, in particular the ability to subscribe to message based on wild cards.
In addition, the MQTT Add-on of the product PcVue implements a series of mechanisms to make topic handling and configuration even easier:

  • The ability to define a global generic root topic, so that variable mapping consists in setting the relative topic leaf only.
  • The ability to define a variable mapping with an absolute topic name (full topic name), and bypass the global generic root topic. This makes it possible to handle messages that are exceptions because they do not follow a general topic naming system applied across your MQTT system.
  • The ability to map variables to topics one-to-one. This makes variable mapping on topics very easy if the MQTT topic naming system is aligned with the variable naming system.

Topics names, wild cards and GenericRootTopic are case sensitive.

Wild cards

When a client subscribes to a topic, it can subscribe to the exact topic of a published message or it can use wild cards to simultaneously subscribe to multiple topics. There are two different kinds of wild cards. Please note that only one wild card can be used at a time.

  • Single level wild card '+' - A single-level wild card replaces one topic level. The plus '+' symbol represents a single-level wild card in a topic name.
    Example - An MQTT client subscribing to topic BuildingA/Floor7/+/Temperature will receive all Temperature-related messages for all Zones of Floor7 in BuildingA.

  • Multi-level wild card '#' - The multi-level wild card covers 1 or more topic levels. The hash '#' symbol represents the multi-level wild card in the topic name. For the broker to determine which topics match, the multi-level wild card can only be placed at the end and preceded by a forward slash (/).
    Example - An MQTT client subscribing to topic BuildingA/Floor7/ConferenceRoom/# will receives all messages for the Zone ConferenceRoom of Floor7 in BuildingA.

The use of wild cards to address topics only makes sense in the context of MQTT Subscribe messages. It does not makes sense in the context of MQTT Publish.

Generic root topic

The Generic root topic mechanism was designed to take advantage of a naming scheme with a certain level of genericity for topic names, similar to recommendations related to designing the variable naming scheme (branches and leaves). It is best illustrated using an example.

If you want to link 3 variables with the following topics:

  • BuildingA/Floor7/ConferenceRoom/Temperature
  • BuildingA/Floor7/ConferenceRoom/Humidity
  • BuildingA/Floor7/ConferenceRoom/Light

If you are interested in processing and mapping all data related to this conference room, a good practice is to use the GenericRootTopic parameter (defined in svmgrMQTT.ini) with the following content, to predefine a "default branch" applicable to topics in the variable mapping:

  • BuildingA/Floor7/ConferenceRoom/#

Then you can specify only the "leaf" topics names in the variable mapping as follows:

  • Temperature

  • Humidity

  • Light

Absolute topic name

If you want to link another variable, which is not using the same root, you can bypass the GenericRootTopic parameter using '@' before the whole topic name you want subscribe to as follows:

  • @BuildingA/Floor7/Hall/Light

These topics are called absolute topics. If no GenericRootTopic parameter is defined in the svmgrMQTT.ini file, all topic names used must be preceded by “@” symbol. The quality of service of the subscription is the highest quality of service of all leaf topics configured on a broker.

Generic root and Absolute topic names for Publish

The GenericRootTopic is also used for published variables. For instance, if the GenericRootTopic parameter is defined with the following:

  • BuildingA/Floor7/ConferenceRoom/#

And if the variable topic name defined in the attribute is:

  • SetAC

The variables will be published on the topic:

  • BuildingA/Floor7/ConferenceRoom/SetAC

As for a subscribed variable, if you want to bypass the GenericRootTopic and use an absolute topic name, then, the topic name of the variable must be preceded by “@” in the mapping of the published variable.

Using the variable name as topic name automatically

Because the MQTT topic naming scheme and the variable naming scheme concept are very similar, there is a very easy way to configure the variable mapping when both naming schemes match. It consists in using the keyword VarName as the SubTopicName or PubTopicName.

If prefixed with the ‘@’ character, the full variable name will be used as topic name; if not prefixed with an '@', the GenericRootTopic is appended at the beginning of the topic name.

Example:

If the GenericRootTopic is defined as CampusNorth/# in the svmgrMQTT.ini file.

For a variable which full name is BuildingA.Room3.Temp, defined with the following MQTT mapping parameters:

  • MQTT|Generic|MyBroker|SUB|0|VarName
    • The MQTT Add-on subscribes to messages having the topic CampusNorth/BuildingA/Room3/Temp that matches the GenericRootTopic concatenated with the variable name.
  • MQTT|Generic|MyBroker|SUB|0|@VarName
    • The MQTT Add-on subscribes to messages having the topic BuildingA/Room3/Temp that matches the full variable name.

The same applies to Publication. For a variable which full name is BuildingA.Room3.Temp with the following MQTT mapping parameters:

  • MQTT|Generic|MyBroker|PUB|0|0|VarName
    • The MQTT Add-on will publish messages with the topic CampusNorth/BuildingA/Room3/Temp
  • MQTT|Generic|MyBroker|PUB|0|0|@VarName
    • The MQTT Add-on will publish messages with the topic BuildingA/Room3/Temp
